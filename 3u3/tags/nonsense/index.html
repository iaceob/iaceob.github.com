HTTP Server

Apache Web Server 应该是 Apache 最知名的项目, WEB2.0 兴起的时代 PHP 因为语法简单开发且部署相对于当时的 JAVA C# 方便很多, 而且在开发效率上也明显快于前两者, 因此受到宠幸, 但是 PHP 就自身而言是不能独立运行的, 得依赖一个 WEB 服务器, Apache 也刚好是那个时候非常成熟的解决方案, 因此 Apache 也就顺理成章的随着一起被使用的越来越多, 各个主机商, 默认自带 Apache 服务器, 随后吵的很火的 LAMP 环境.
但是 Apache 绝不是靠着 PHP 才走到今天, 不过如果没有 PHP 那 Apache 也不可能像如今这样.
那就拿 Apache, PHP 结合来说一下吧
首先来看看要怎么来配置让 Apache 支持 PHP, Apache,PHP,MySQL的安装，配置
是不是真的太特么长了!!! 好麻烦的啊!!! 各种配置文件, 而且各个配置文件位置还不一样, 修改 Apache 的配置文件支持的时候还得在那很长很长的配置文件里面找那几行来开注释. 哦…
你说, 不是 LAMP 环境嘛, 你以为 Linux 下配置就简单了??? 换汤不换药而已, 而且相较于 Linux 更完善的权限管理以及 Linux 其自身就凌乱的配置来说, 配置 Apache 会更复杂, 很有可能你安装 Apache 后都不知道配置文件到底特么放在什么地方, 但如果是 Windows 通常肯定会放在安装目录.

关于 LAMP 的一键安装, 不在本文考虑范畴, 而且一键安装工具就是因为做这个安装方式的人发现实在太烦了, 才会想办法让以后不用这么烦.

Commons

这个项目很有可能是 JAVA 世界中应用最广泛的工具包, 这个工具包非常全, 几乎把基础的数据库操作都搞定了, 像是 IO, DB, 文件上传 等等 甚至你想用 JAVA 写一个命令行程序, 这个工具包都提供了参数解析.
如何使用可以参考: 编写更少量的代码：使用apache commons工具类库
可以说非常的实用.

然而
让我们来看一看 帕雷托法則 相信看完后就会觉得, 喔 好像确实是这样, 或者还没理解是想要表达什么意思, 简单来说, Commons 包你能用到的功能顶多就只有 20% 而已, 相信我, 这是肯定的, 然而为了这 20% 却要让你的项目多 80% 用不上的东西.

其实这也不是太大的问题吧, 毕竟那 20% 是刚需, 如果没有则有可能会重造一个轮子, 还不一定是安全的轮子.
所以说, Commons 是一个重量级的库, 实际选择中斟酌使用, 量需而行.

Cordova

Cordova 可以说是时代的时代的产物, 这么说可能不合适, 因为几乎任何事物的出现都是时代的产物, 其工作原理就是将 HTML 应用打包成一个近似于原生的 APP 促进了移动互联网的发展, 降低了 APP 的开发难度, 而且能把以及很完善的 WEB 技术带入到 APP 的开发中.
然而, 也只是用过而已, 只是将官方的例子稍微改了一下增加一个页面交给 Cordova 去打包, 打包后也确实能安装成功.
没有发现太大的坑.

Groovy

Groovy 其实也是个很成功的项目, 很好的动态特性, 而且也天生就支持所有 Java 代码, 包括非常成熟的库, 就比如 Commons, 然而自出生到现在一直是不温不火, 能见到的最多的应用场景就只是在配置文件 (logback 配置文件支持 Groovy, 以及 gradle 包管理工具), 或者是插件的开发上 (pancake-blog) 的插件就是用 Groovy 实现的, 确实是能很好的胜任.
如果要对比的话, 同类型的与 Python 相比, 可能因为 JVM 自身的问题可能在选择上会更倾向于 Python
但是同样是跑在 JVM 的 Scala 都已经有出现很多成熟的项目了, 而且 Scala 是被各种鼓吹 (虽然不知道是不是真的 当然我在扯淡, 王垠的博客中曾有说过 Scala 等各种语言的问题, 有兴趣可以看一看), 反观 Groovy 仍然不温不火.

资源文件夹

有没发现不统一的地方?
明明都是在配置路径, 引入资源用 resources>(resource>directory)*2 这种强格式要求来实现. 而 war plugin 中排除路径确实靠内容中的格式要求, 而不是 configuration>packagingExcludes>packingExclude*2 这种统一的方式来实现.
你会说这两个本来就不是一种功能当然有可能不一样, 那好就认了.
如果多试几个插件, 就会发现, 每个插件的这中配置都是有可能不同方式的, 有个的靠 XML 标签强格式要求, 而有些又是靠内容格式约定来实现 = =

总结

总算说完了, 就总结一下吧.
Apache 确实是个很出色, 且很牛逼的代码项目聚集的地方, 很多项目也一直在跟着互联网的变换而进行迭代, 来满足日益变化的需求.

但是其作风不统一, 很学院派, 基本上可以说每个项目的配置都非常的复杂 繁琐, 不能做到简单易用, 就例如 Http Client, 这个上面还忘记提了, 其实就简单的实现逻辑而已, 使用 Http Client 一个最简单的 GET 请求一个地址而已, Http Client 要用十几行代码来完成, 创建各种对象, 而且在 Http Client 甚至是包括 Apache 的其他项目中, 因为版本迭代而造成了很多的老版本的接口调用被证实是不安全或者说有一个更好的方式实现, 而将老接口用 @Deprecated 来告知用户, 就得花很多时间来去找到底替换成了什么接口, 因为有些新的实现方式完全和老接口的实现方式不一样 …
反正就是乱象从生, 况且在中文网络中, 充斥者抄袭等风气, 去查找这类问题很多一部分结果站点都是那种那蜘蛛去爬的 SEO 做好, 放一堆广告 … 再有中文网络中这种新的技术或者说接口变动等信息更新的很晚.

Apache 开发管理的项目, 特别喜欢配置, 各种配置, 只要是能配置的东西全拿出来搞, 其实这没什么, 反而就程序而言这样是好的, 但是要吐槽的是, 怎么那么喜欢用 XML !!! 很乱的啊 …
像以前 ssh 联合框架的整个配置就能搞死人, struts 就有一份功劳, 现在的 sping, spring mvc, mybatis 组合仍然是这个尿性 …

  • 信XML，得永生！
  • 信XML，得自信

Apache 基本都是在生产重量级的产品, 当然也就意味着去实现的是更高级的层面, 但是, Apache 的某些设计方式绝不是良性的, 应该更简化一些操作方式, 将复杂的事情简单化, 而不是说设计一个功能很牛逼的项目但是却提供了屎一样的 API.

改天就来说说 spring, spring mvc, mybatis

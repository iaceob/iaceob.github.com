Hackpad 部署

其實這篇就算不寫想要部署的你也應該能夠完成, 因為只需要看官方文檔就已經完全能夠成功部署, 那麼就當作記事吧.

首先, 如果你還不知道 Hackpad 是何物, 建議到 Hackpad 官網 試用.

然後 Hackpad 源碼地址: Hackpad GIT .

部署也很簡單, 參見 INSTALL, 完全按照文檔中的流程操作即可.
該安裝文檔所贅述的安裝方式是在 Mac 環境下的部署方式, Linux 上部署流程也是相同.

  1. 首先安裝 JDK 與 Scala, 只需要在其官網下載, 並且配置環境變量即可.

  2. 關於 MySQL 可以通過 apt-get yum 等方式安裝, 視你的操作系統.

  3. 修改 bin/exports.sh 將其中的 JAVA 以及 Scala 路徑修改成你部署的環境, 需要注意 Scala 的配置中

  4. 此處的路徑, 可能在 Mac 中 Scala 有 libexec 這個路徑, 如果你是通過下載配置環境變量的方式, 在 Linux 中, 是沒有 libexec 路徑的, 直接 /lib 即可.

  5. 執行 bin/build.sh

  6. 執行 contrib/scripts/setup-mysql-db.sh 創建數據庫

  7. 複製 etherpad/etc/etherpad.localdev-default.properties 為 etherpad/etc/etherpad.local.properties
    且修改複製的文件將一些配置信息改為你的配置

  8. emoji 支持
    下載 https://github.com/github/gemoji/tree/master/images/emoji/unicode 圖片 並複製到 etherpad/src/static/img/emoji/unicode/ 即可

  9. 執行 bin/run.sh 啟動, 訪問 http://localhost:9000 查看部署成功的 Hackpad

  10. 關於 Chrome 無法登錄問題
    編輯 infrastructure/framework-src/modules/global/response.js 文件
    將如下行註釋 https://hackpad.com/ep/pad/static/lM3FdARmrGD

  11. 同步 Dropbox 以及 Google Facebook 賬戶登錄
    如果你在天朝, 那就別想了, 否則就在 etherpad/etc/etherpad.local.properties 相應的配置上填入相應值 (我沒試).

題外:
MySQL 的安裝, 也可以通過手工安裝方式, 參照 MySQL5.6源码安装 該系列, 絕壁可以成功的方式.

当然, 我就是在扯淡

准备出一个系列的扯淡文章, 对自己的定向思维的进攻吧.

首先要说的就是 Apache

Apache 有多出名了, 相信绝大部分人都知道, 某军事频道某 “菊座” 时不时就能提起来, 就算不知道 Apache 到底是个什么东西, 也绝壁听过这么个名号.
那么就来扯一扯 Apache.

首先, 先到 Apache 官网来看看 .
欸, 竟然改版了, 不过没有关系, 让我们将滚动条拉到最下方, 对, 是不是有很多英文单词, 数一数你能认识几个?
本人不才, 只认识下面这些

哎哟 我自己都不敢相信尽然还能认识这么多 (记住, 只是认识而已), 那么就一个个来说

HTTP Server

Apache Web Server 应该是 Apache 最知名的项目, WEB2.0 兴起的时代 PHP 因为语法简单开发且部署相对于当时的 JAVA C# 方便很多, 而且在开发效率上也明显快于前两者, 因此受到宠幸, 但是 PHP 就自身而言是不能独立运行的, 得依赖一个 WEB 服务器, Apache 也刚好是那个时候非常成熟的解决方案, 因此 Apache 也就顺理成章的随着一起被使用的越来越多, 各个主机商, 默认自带 Apache 服务器, 随后吵的很火的 LAMP 环境.
但是 Apache 绝不是靠着 PHP 才走到今天, 不过如果没有 PHP 那 Apache 也不可能像如今这样.
那就拿 Apache, PHP 结合来说一下吧
首先来看看要怎么来配置让 Apache 支持 PHP, Apache,PHP,MySQL的安装，配置
是不是真的太特么长了!!! 好麻烦的啊!!! 各种配置文件, 而且各个配置文件位置还不一样, 修改 Apache 的配置文件支持的时候还得在那很长很长的配置文件里面找那几行来开注释. 哦…
你说, 不是 LAMP 环境嘛, 你以为 Linux 下配置就简单了??? 换汤不换药而已, 而且相较于 Linux 更完善的权限管理以及 Linux 其自身就凌乱的配置来说, 配置 Apache 会更复杂, 很有可能你安装 Apache 后都不知道配置文件到底特么放在什么地方, 但如果是 Windows 通常肯定会放在安装目录.

关于 LAMP 的一键安装, 不在本文考虑范畴, 而且一键安装工具就是因为做这个安装方式的人发现实在太烦了, 才会想办法让以后不用这么烦.

Commons

这个项目很有可能是 JAVA 世界中应用最广泛的工具包, 这个工具包非常全, 几乎把基础的数据库操作都搞定了, 像是 IO, DB, 文件上传 等等 甚至你想用 JAVA 写一个命令行程序, 这个工具包都提供了参数解析.
如何使用可以参考: 编写更少量的代码：使用apache commons工具类库
可以说非常的实用.

然而
让我们来看一看 帕雷托法則 相信看完后就会觉得, 喔 好像确实是这样, 或者还没理解是想要表达什么意思, 简单来说, Commons 包你能用到的功能顶多就只有 20% 而已, 相信我, 这是肯定的, 然而为了这 20% 却要让你的项目多 80% 用不上的东西.

其实这也不是太大的问题吧, 毕竟那 20% 是刚需, 如果没有则有可能会重造一个轮子, 还不一定是安全的轮子.
所以说, Commons 是一个重量级的库, 实际选择中斟酌使用, 量需而行.

Cordova

Cordova 可以说是时代的时代的产物, 这么说可能不合适, 因为几乎任何事物的出现都是时代的产物, 其工作原理就是将 HTML 应用打包成一个近似于原生的 APP 促进了移动互联网的发展, 降低了 APP 的开发难度, 而且能把以及很完善的 WEB 技术带入到 APP 的开发中.
然而, 也只是用过而已, 只是将官方的例子稍微改了一下增加一个页面交给 Cordova 去打包, 打包后也确实能安装成功.
没有发现太大的坑.

Groovy

Groovy 其实也是个很成功的项目, 很好的动态特性, 而且也天生就支持所有 Java 代码, 包括非常成熟的库, 就比如 Commons, 然而自出生到现在一直是不温不火, 能见到的最多的应用场景就只是在配置文件 (logback 配置文件支持 Groovy, 以及 gradle 包管理工具), 或者是插件的开发上 (pancake-blog) 的插件就是用 Groovy 实现的, 确实是能很好的胜任.
如果要对比的话, 同类型的与 Python 相比, 可能因为 JVM 自身的问题可能在选择上会更倾向于 Python
但是同样是跑在 JVM 的 Scala 都已经有出现很多成熟的项目了, 而且 Scala 是被各种鼓吹 (虽然不知道是不是真的 当然我在扯淡, 王垠的博客中曾有说过 Scala 等各种语言的问题, 有兴趣可以看一看), 反观 Groovy 仍然不温不火.
这是为什么?

Maven

Java 世界中使用最多的包管理工具, 没得说, 对于 Java 的各种包管理依赖确实是个好工具, 省去了原始的手工一个个添加包, 还要考虑版本等问题, Maven 就能很方便的管理好, 而且还有很多的插件, 从编译到打包甚至跟多功能.
but 各种配置又尼玛很乱的啊… 举个例子:

资源文件夹

有没发现不统一的地方?
明明都是在配置路径, 引入资源用 resources>(resource>directory)*2 这种强格式要求来实现. 而 war plugin 中排除路径确实靠内容中的格式要求, 而不是 configuration>packagingExcludes>packingExclude*2 这种统一的方式来实现.
你会说这两个本来就不是一种功能当然有可能不一样, 那好就认了.
如果多试几个插件, 就会发现, 每个插件的这中配置都是有可能不同方式的, 有个的靠 XML 标签强格式要求, 而有些又是靠内容格式约定来实现 = =

Shiro

RBAC 权限管理方式的 Java 实现版本, 功能很强大, 完全能根据 Shiro 的机制来实现权限的控制.
然而在以前的文章, 我就有说过 RBAC 权限管理的弊端, 不能灵活控制, 详见老博客 (目前老域名暂未申请, 服务器也因为性能考虑, 将 PHP 给禁止了, 静态化暂时还没做 = =).

Struts

J2EE 项目中控制器 (路由) 管理工具, 其第一个版本, 因为设计上的问题, 导致使用的较少, 长说的 struts 通常是 struts2. Struts1和Struts2的区别

已经很久都没有在用 strut2 了, 想想啊, 从最初开始学习 Java EE, 就是学的 struts2, 而且是 ssh 结合的那种, 在我记忆中最深的感受就是配置非常的繁琐, 每个 route 都得在 struts.xml 中配置好, 并且指到指定的 Action 类中.

Subversion

总结

总算说完了, 就总结一下吧.
Apache 确实是个很出色, 且很牛逼的代码项目聚集的地方, 很多项目也一直在跟着互联网的变换而进行迭代, 来满足日益变化的需求.

但是其作风不统一, 很学院派, 基本上可以说每个项目的配置都非常的复杂 繁琐, 不能做到简单易用, 就例如 Http Client, 这个上面还忘记提了, 其实就简单的实现逻辑而已, 使用 Http Client 一个最简单的 GET 请求一个地址而已, Http Client 要用十几行代码来完成, 创建各种对象, 而且在 Http Client 甚至是包括 Apache 的其他项目中, 因为版本迭代而造成了很多的老版本的接口调用被证实是不安全或者说有一个更好的方式实现, 而将老接口用 @Deprecated 来告知用户, 就得花很多时间来去找到底替换成了什么接口, 因为有些新的实现方式完全和老接口的实现方式不一样 …
反正就是乱象从生, 况且在中文网络中, 充斥者抄袭等风气, 去查找这类问题很多一部分结果站点都是那种那蜘蛛去爬的 SEO 做好, 放一堆广告 … 再有中文网络中这种新的技术或者说接口变动等信息更新的很晚.

Apache 开发管理的项目, 特别喜欢配置, 各种配置, 只要是能配置的东西全拿出来搞, 其实这没什么, 反而就程序而言这样是好的, 但是要吐槽的是, 怎么那么喜欢用 XML !!! 很乱的啊 …
像以前 ssh 联合框架的整个配置就能搞死人, struts 就有一份功劳, 现在的 sping, spring mvc, mybatis 组合仍然是这个尿性 …

  • 信XML，得永生！
  • 信XML，得自信

Apache 基本都是在生产重量级的产品, 当然也就意味着去实现的是更高级的层面, 但是, Apache 的某些设计方式绝不是良性的, 应该更简化一些操作方式, 将复杂的事情简单化, 而不是说设计一个功能很牛逼的项目但是却提供了屎一样的 API.

改天就来说说 spring, spring mvc, mybatis

《甲骨文》 美國之音 節選

威利和南西提出了申請, 他們的政治背景不是很出色: 兩人都沒有加入共產黨, 也不得幹部特別偏愛. 除了其他一些小違規外, 兩人的戀愛還曾遭過批評. 畢業在即, 他們卻還沒得到上面的回覆.

最後威利決定採取行動 – 他沒有文任何人的建議. 多年後他解釋說, 當時他只是聽從自己的 「第六感」.一天晚上, 他偕同南西到當書記家. 當書記向來不怎麼友善, 威利也不太喜歡他; 但那晚, 他笑著請威利進門, 他稱讚說, 玉環聽起來像是個經濟前景看好的地方, 不過, 浙江很遠, 轉送文件到那兒可不是那麼簡單的事情.

威利跟他解釋, 往東發展是他最想做的事, 他掏出一個信封袋, 放在書記面前的矮茶几上, 說: 「請幫忙.」

「我試試,」書記說. 然後他送那對年輕人到門口. 誰也沒提一句關於那信封袋的話.

另外一名職員負責畢業生的工作編遣, 威利和南西到他的住處重施故技. 每個信封袋裡面裝著五百元人民幣, 總數等於威利父親半年的收入. 那也是他有生以來第一次賄賂職員幹部.

畢業前不久, 學校通知威利和南西, 他們的檔案已送到他們申請的東部.

Request InputStream 首先獲取

事情是這樣, 因為需要文件上傳功能, 就按照以前使用過的方式在控制器中使用 getFile 獲取上傳的文件, 但是這次報了如下錯誤

從報錯信息中沒有看出是什麼地方的錯誤, 起初懷疑是在文件上傳表單中未設置 multipart/form-data; 而導致無法獲取文件, 遂排查, 但發現明明是有設置, 並且在 Request Header 中確實是 multipart/form-data; , 工程中關於文件上傳的依賴包也已經引入.
那問題就不是發生在這一步, 後來在控制器中輸出 request.getContentLength 發現是有長度, 那麼也確實文件是有被上傳的.

那麼問題就是發生在 getFile 方法中, 定位到錯誤地方, 進入調試最終定位的錯誤的代碼是 org.eclipse.jetty.server.Request

在使用 getFile 獲取文件時, 發現 this._inputState 的值是 2, 所以就不滿足這個 if 判斷, 而直接拋出 IllegalStateException 異常.

所以就得找 this._inputState 是在何處被賦值為 2 .

後在 Request 類中的 getReader 方法中發現

原來在此處! 問題就是發生在這裡, 因為這次項目中提供的接口中參數傳遞都是使用 JSON 方式, 為了便於在控制器中解析數據, 遂加了一個參數解析的過濾器, 將獲取到的 JSON 字符串轉換為對象以便於在控制器中取值, 獲取 JSON 時就調用了 getReader 方法獲取流中的 JSON 數據, 這是執行在全局的過濾器, 所以在後續的控制器中通過 getFile 也就是 getInputStream 中獲取上傳的文件時就會拋出前面提到的錯誤.

so, 文件上傳的控制器中取消參數解析的過濾器即可, 獲取 getFile 必須是在獲取參數之前的, 否則將無法獲取.

起點

翻翻日曆(應景), 臥槽都特麼四月了 !!!!
這一年都特麼到了三分之一了 !!!!
從來沒感覺過能有這麼快的啊 !!!!

日常還是日常, 似乎仍然沒有收拾好心情來做事.

這期間值得一提的事情是, 第一本主動閱讀的書讀完了 《消失的江城》 從去年 (2015) 直到前幾天清明節假期回家路上讀完, 說來也巧, 書中有一章節是 春節 , 而剛好是我在春節回家途中開始閱讀的, 又是春天 又剛好是在這剛步入到春季時閱讀, 剛好恰到好處的感受了一番, 確實是一本有意思的書籍.

so, 下一個書單起點開始了 《甲骨文》

然後,

開始準備向後面開始準備吧, 可能不久就會離開吧 ╮(╯_╰)╭ 仍未發現哪裡是可以停留的地方, 流浪吧.

這幾次回家都和爺爺深談了好久, 談以前的家鄉, 談以前爺爺小時候, 用滿滿的好奇去問爺爺很多問題,

比如以前的黃尾鎮, 容我畫張圖表達試一試

原諒我放蕩不羈的畫風

故事是這樣的, 爺爺是在 1950 年出生的共和國第一代國名, 那個時代的黃尾大概就像這樣 (雖然似乎直到現在也基本還是這樣), 在河的這面說是城牆 (聽的我一陣驚訝, 我說怎麼現在城牆沒了了!!), 爺爺那會還和小夥伴過城門到河裡去游泳.
在 60 年還是哪一年來著, 發大水把整個城牆給沖走了, 整個黃尾鎮的房子都被沖走了, 我是各種不相信啊, 竟然這麼誇張, 而且城牆還給沖走了, 那可是城牆欸.
再細一說, 原來所謂的城牆是攔河壩, 比地稍微高一些, 對嗎 這樣才合乎情理啊, 本來家鄉這都可以算得上是荒無人煙的地方了, 怎麼會又城牆這麼高大上的東西, 在那沖毀了之後, 這個攔河壩又重修了, 一直到現在, 現在這個壩比以前還要長

然後是這座橋, 容我在畫一張圖

當然這麼大的水肯定也是把整座橋給沖毀了的 (未知是否那次沖毀後重建的新橋), 以前的橋只有一人能走的寬度, 十三節, 每節多長來著 (說著我給忘了, 注意! 圖上的也是十三節), 那時候還有乘船過河的, 河道很寬, 還說這座橋有過人走時不慎摔到河裡死人的.
之後就是這座新建的橋了, 橋很寬, 得有 7 米吧應該 (應該差不多, 可能不止), 石拱橋, 三拱, 但拱得又超過 20 米 (感覺而已), 佩服那時勤勞的人, 修了這樣一座橋, 真的, 這座橋很漂亮, 全部都是石結構, 有了這座橋之後, 沒有了划船的人, 也見不到船了, 當然就算有船也已經划不過去了, 並不是沒人會划船, 而是共和國決定在河道的上方修建了一個水電站, 以及不知道什麼原因河道的水也是慢慢的變少了, 自此之後, 正常情況下, 這條河的河水的水位就下降了, 就連河中間的河床都基本浮現了出來, 只有很小的一部分流淌這湍急的河水. 再之後, 這幾年在河道上游, 水庫下游又給修建了一個旅遊區, 又給加了一道水壩, 水量又一次減少, 我覺得這似乎並不是一件好事, 大好環境就被如此人為的給破壞了, 已經回不到最初的河流了.

被破壞的又何止只是這條河流了.

原本堅挺有力的山峰, 在山頂又被某個開發商在那採掘石頭, 採掘後的石頭又直接在河邊二次加工, 加工後的乳白色廢水排放在河里, 在往下走一個支流上方是一個鐵礦加工區, 排出黃色廢水, 結合後黃白相間向下游流去, 甚至那鐵礦加工區還直接改變原有彎曲河道從一個伸出的半島直接開口, 而那彎曲的部分放置礦渣. 現在鐵礦開採也沒有在繼續了, 這個鐵礦留給家鄉的就三點:

  • 一夜暴富的移民戶
  • 未被善後的採礦坑
  • 被礦渣填平的河道

各種被破壞的現狀.

說遠了

然後就和爺爺說了句, 以前真好, 回應我的是一句:

「好個屁, 以前都吃不飽, 那個時候很苦.」

當然, 我已經很難體會那時候了, 只能說略有點感受吧, 至少在我的記憶中, 以及我所經歷的見聞, 以前確實是很清貧. 但是以這樣的環境犧牲為代價是可取的?

這次下來還聽到爸爸這麼說了:

「以前看山上到處都是花, 能問到各種花香, 而現在山上的花都少見了, 映山紅 蘭草 都沒了」

在這次之前的一次談話哩, 說了一說以前爺爺小時候發生的事情.

因為爺爺是 1950 年出生的, 所以了, 共和國成立之初開始摸石頭過河做過的路爺爺可以說是個盡職的摸石頭的人, 練過鋼 當過紅衛兵 吃食堂 各種計劃經濟票據, 談到批鬥一事, 說是那個時候整天不做事情, 就是跟著隊伍從這跑到那響應號召. 而且有真正見過人被批鬥, 說是很殘忍以至於我想繼續問下去但是卻想了好久才慢慢道出來, 這也許是最難以回首的經歷吧, 談過之後才瞭解到原來處在身邊的人當時都有這樣的經歷甚至是被批鬥的一方.
政治甚至能影響到這樣的偏僻小山村, 在那個道路不通, 消息敝塞的時代, 真真切切的印象了那一代人.

好吧, 就說這些了, 這也不適談的太深. ╮(╯_╰)╭

基本上了, 每次都是我先終止了話題, 沒有在繼續談下去, 也不知是在某個時刻產生的想法, 很想已書籍的形式來記錄這些, 但是又無從下手, 關於那個年代的故事我們現在的人已經知之甚少了, 或者說能知道的渠道基本都不存在, 而像爺爺奶奶這樣年齡的人又會慢慢的少去, 記憶改如何留存下去了 ? 可惜我並不很擅長寫文章.

好啦, 四月開始了, 新的起點又來了.

寶寶的生日也快到了.

生活總是要面對的, 最後說說最近學的新知識吧

沒錯 還是寶寶教的 XD

然後

勉強能認識這些 _(:3 」∠ )_

學習繼續, 調情睡覺.

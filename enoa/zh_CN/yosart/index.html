Yosart

Yosart

Yosart 是一个基于 Repeater 的 Web 框架, Yosart 的涉及极其精简, 其核心只是提供基本的 Web 服务的接口, 剩下的交给插件或扩展去实现.

结构图

还记得 Repeater 的结构图吗? Yosart 是 Repeater 的上游高级应用, 实现了 Repeater 提供的 EoxAccessor 接口, 增加了路由, 渲染等功能.

图中右侧便是 Yosart 的结构图, 接入到 Repeater 体系中.

首先来看看 Yosart 中重要的几个接口和类

  • OysartAccessor

    Yosart 访问器, 提供给 Repeater, 用于接收请求

  • OyResource

    Yosart 所有控制器分析后的资源, 包括 YoAction 以及 YoControl

  • YoExt

    Yosart 扩展接入接口

  • YoPlugin

    Yosart 插件加入接口

核心

Yosart 核心部分均在 io.enoa.yosart.kernel 包下, kernel 下的所有包, 以下核心便是 Yosart 所做的事, 抛却这些核心, 剩下的全部交给 插件 或 扩展 进行处理.

anno

注解包, Yosart 提供的注解, 仅有一个 Action 注解, 该注解用于申明接口 uri 地址, 以及允许的 http method.

data

数据承载实体类

ext

关于 Yosart 的扩展, 需要明白, 所有的扩展均来自于 YoExt, 不过 YoExt 并不是被直接实现的, 而是通过具体不同的接口实现, 例如 渲染扩展 (YmRenderExt) 路由扩展 (YoRouterExt) 等, 详细见 扩展

http

Yosart 基于 Repeater , 但是 Repeater 的 Request 仅仅是简单的 http 请求信息, 但是对于 Web 框架而言这些是不够的, Yosart 支持 Restful 风格的 API, 这些 Repeater 并没有做, 且 Repeater 也未提供将请求参数转换为实体的方法, http 包下是 Yosart 对 Repeater 的 Request 的高级封装.

mount

mount 中只有一个 OyMountd 类, 在正常的业务逻辑开发中是用不到的. Yosart 提供一个信息输出的页面 (后续会说道). 而 OyMountd 是将其他组件的信息挂载到 Yosart 中.

render

所有渲染扩展最终渲染的返回接口 YoRender

resources

Action 资源包, YoAction , YoRouter 以及 YoControl 均定义再此, 也包含所有 Action 资源的分析, 最终处理成 OyResource

Resp

Yosart 除了核心包之外, 还提供了 Response 构造器, io.enoa.yosart.resp 包下, 存在两种构造器

  • Resp
  • Renderer

区别是, Renderer 所构造的内容都是通过渲染扩展进行渲染的内容, 而 Resp 可以构造渲染内容之外, 还支持 Response 相关值的设置, 包括 Header Cookie 等.

Restful

Yosart 支持 Restful 风格接口, 接口 uri 定义遵循以 : 表明参数名, 例如以下案例

GET  /api/user/:id
PUT  /api/user/:id

即可通过 Request 中 variable("id") 的方式获取.

插件 & 扩展

关于插件和扩展的区别, 早些年在 firefox 贴吧中有一句话描述的很好 扩展负责扩开来, 插件负责插进去.

插件

Yosart 插件, 只需要实现 YoPlugin 接口即可, 启动时, 加入到 Yosart 中.

扩展

Yosart 扩展目前支持的有

  • YmAssetsExt

    静态资源扩展

  • YmBeforeActionExt

    Action 前置扩展, 用于在运行 Action 前运行的操作

  • YmBootHookExt

    启动钩子, 会在 Yosart 启动之前调用的扩展

  • YmExceptionExt

    用于在调用 Action 出现异常的情况下运行的扩展

  • YmRenderExt

    渲染扩展

  • YmRouterExt

    路由扩展

需要注意的是, 这里理出的扩展并非都是 Yosart 直接支持, Yosart 直接支持的有 YmAssetsExt, YmRenderExt, YmBootHookExt 以及 YmRouterExt.

YmBeforeActionExt 和 YmExceptionExt 视选择的 YmRouterExt 是否支持, Yosart 无法控制, 如果选择的路由扩展不支持 YmBeforeActionExt 和 YmExceptionExt 则无法使用.

自定义扩展需要注意 weight 方法, 该方法的含义是此扩展的权重, 权重将影响扩展的运行顺序, 可取值范围 1-9 之间浮点数, 0-1 以及 9-10 为 Yosart 保留权重. 默认会随机生成, 无需关注.

使用

Yosart 自带 EyRouterExt 路由扩展, 因此下面描述的使用方式均是 EyRouterExt 所支持的.

一个简单的 Yosart 使用案例.

Yosart.createServer()
  .handle(Method.GET, "/", (req) -> Resp.with(req).render("yosart").end())
  .listen();

Yosart 默认搭载的 Web 服务器是 NanoHttpd, 默认监听端口 9102, 只需在工程中引用 Repeater 以及 NanoHttpd 即可.

是的, 这个很简单, 接下来看一个很完善的使用案例

Yosart.createServer()
  .config(this.config())
  .repeater(new YasartRepeaterFactory())
  .provider(new NanoHTTPDProvider())
  .log(new Slf4JLogProvider())
  .plugin(new JsonPlugin(new EoJsonProvider()))
  .plugin(this.activeRecordPlugin())
  .plugin(this.beetlSQLPlugin())
  .plugin(this.mybatisPlugin())
  .plugin(new ConfPlugin("https://github.com/mstine/config-repo.git"))
  .ext(new JsonRenderExt())
  .ext(new TemplateRenderExt(TEMPLATE.ENJOY.factory, TEMPLATE.ENJOY.basePath, TEMPLATE.ENJOY.suffix, new HtmlCompressorProvider()))
  .ext(new BeaExt())
  .ext(new YemgrExt())
  .context("/example")
  .assets("/assets", Paths.get(EnvKit.get("java.io.tmpdir")), true)
  .handle(Method.GET, "/", (req) -> Resp.with(req).attr("req", req.data()).renderTemplate("info").end())
  .handle(YosartControl.class)
  .handle(new WorkRouter())
  .handle("/yosart", (req) -> null)
  .handle("/party", PartyControl.class)
  .handle("/corpse", new ZombieRouter())
  .handle("/download", (req) -> Resp.with(req).renderFile(Paths.get(String.format("%s/download/file.txt", PathKit.getRootPath()))).end())
  .handle("/db", DbControl.class)
  .handle("/actionanno", new YoAction() {
    @Before({ActionAnnoValid.class, PartyGoHandler.class})
    @Override
    public Response handle(YoRequest request) {
      return Resp.with(request).render("action anno").end();
    }
  })
  .listen(9102);

案例代码可通过源码 /enoa-example/example-yosart 获取

这个案例中涉及到了很多, 插件 扩展, 这些不属于 Yosart 要介绍的部分, 去查阅相关插件或扩展的文档. 这里主要介绍与 Yosart 实现有关的部分.

在说路由之前, 先看以下 Yosart 启动接口, 该接口是 Yosart 最中的地方, 所有插件, 扩展 路由都是在这里注册. 下面列出 Yosart 所有方法

  • provider

    选择 Web 服务器提供者, 视需要选择一个提供者.

  • repeater

    Repeater 配置项, EoxAccessor 访问器配置就在此处, Yosart 提供了一个默认的配置 YasartRepeaterFactory, 通常这个是不需要进行配置的, 如果自定义了其中了 accessor 那么也就表示着放弃了 Yosart 提供的请求解决方案, 那么就去看相应的 accessor 是如何工作的.

  • log

    Yosart 日志提供者选择

  • plugin

    添加插件, 插件添加后会马上运行.

  • ext

    添加扩展, 扩展在服务监听前一起运行

  • ssl

    开启 ssl 服务

  • handle

    注册路由

  • name

    应用名称

  • context

    Http context

  • config

    Yosart 配置信息

  • assets

    静态资源配置

  • listen

    服务监听

这里主要说下 handle 以及 assets 方法

handle 路由注册, 供提供了以下几种方式

Yosart handle(Method method, String uri, YoAction action);

Yosart handle(String uri, YoAction action);

Yosart handle(String uri, Class<? extends YaControl> control);

Yosart handle(String uri, YoRouter router);

Yosart handle(Class<? extends YaControl> control);

Yosart handle(YoRouter router);

YoAction

YoAction 提供两种注册方式, 限定 Method 和不限定, 当然这个具体是什么含义视路由扩展如何定义, 默认的路由扩展中, 没有 Method 则表明允许任何 Method 请求, 这个概念在 Control 中也适用.

YoAction 定义方式

new YoAction() {
  @Override
  public Response handle(YoRequest request) {
    return Resp.with(request).render("action").end();
  }
}

也可以使用 Java 8 提供的 lambda 表达式方式

(req) -> Resp.with(req).render("action").end()

YoControl

YoControl 注册, 需要将注册的控制器继承 YoControl/YeControl/ YaControl 类.

之所以提供三种不同 Control 是为了不同的路由实现考虑, 这三种不同的 Control 的区别

CONTROL DESCRIPTION
YaControl 直接实现自 YaResource 接口, 内部存在 Request 以及 Resp
YeControl 继承自 YaControl, 提供将 Request 相同方法而不需要获取 Request 后再获取参数信息
YoControl 继承自 YoControl, 提供 render 系列方法

应用场景不同, 例如默认的路由实现, 因为不支持控制器方法参数解析以及方法的返回值解析, 那么使用 YoControl 方法最合适, render 系列方法则完成了所有渲染功能, 而不用关心方法的参数以及返回值; 另一个路由 yosart-ext-router-anost 则支持方法参数的解析以及方法返回解析, 那么 render 以及 request 的方法已经没什么必要, 完全可以通过方法参数获取 request 参数, 因此可以选择继承 YaControl, 避免开发中 control 的方法污染. 选择哪个取决于所使用的路由扩展, 当然不选择全部用 YoControl 也是可行的.

使用方式

public class IndexControl extends YoControl {
  public void index() {
    super.render("index");
  }

  @Action(method = Method.GET, value = "/action_test")
  public void actionTest() {
    super.render("action test");
  }
}

需要注意, 默认的路由扩展不支持 Control 方法参数解析, 另也无法解析方法的返回值, 当然就算写上也不会报错, 将会收到警告日志, 如果需要支持请选择其他路由实现扩展. (官方携带的 yosart-ext-router-anost 将会支持. 目前尚未施工完毕.)

YoRouter

还有一种注册方式是 YoRouter 路由注册, 该注册方式区别与 YoAction 以及 YoControl, Yosart 会在启动时将 YoRouter 分析成控制器或 Action. 关于 YoRouter 可以看下使用方式

public class WorkRouter extends YoRouter {
  @Override
  public String context() {
    return "/work";
  }

  @Override
  protected void register() {
    super.add(IndexControl.class)
      .add("/holiday", HolidayControl.class)
      .add("/working_day", WorkingDayControl.class)
      .add("/report", new ReportRouter())
      .add("/lazy", (req) -> null);
  }
}

当继承了 YoRouter 之后, 实现两个方法, context 以及 register, context 的意思是该路由所在的 context, 这个 context 需要了解的是前后文的 context.

例如

.handle("/day", new WorkRouter())

那么注册的路由就是

/day/work/holiday
/day/work/working_day
...

如果不需要该 context, 直接留空即可.

register 方式用于注册该路由下的控制器, 同样的也是支持 YoAction, YoControl 以及 YoRouter

assets

另外一个要说的是 assets 方法, 该方法的作用是创建一个静态资源请求的支持.

Yosart assets(String uri, Path path);

Yosart assets(String uri, Path path, boolean showTree);

uri 为静态资源的前缀, path 为静态资源路径, showTree, 表示在请求的路径是目录时是否显示文档列表. 静态资源是走扩展实现的, 默认实现为 EyAssetsExt, 如果想要其他的实现, 实现 YmAssetsExt 接口加入到 yosart 启动中即可.

YoRequest

YoRequest 在 Repeater 的基础上添加 Restful 风格接口 url 参数解析, 以及参数转换 bean 方法.

  • variableNames

    获取所有 uri 参数名

  • variable

    获取参数值

  • bean

    将 Request 参数转化为实体类

其他

session

可能在使用过程中你会发现, Yosart 并未提供 session 的操作功能, 可能无法理解, 很多框架都可以进行 session 操作, 为什么 Yosart 不支持?

首先, 请理解 session 的本质, session 实际上是通过 cookie 去实现的, 创建一个 session 实际上是推送一个 cookie 给前端, 每次请求都会将 cookie 带回来, 服务端通过 cookie 去查找之前设置的 session 信息.

那么, 既然 Yosart 支持 cookie 获取, session 完全可以通过扩展的方式实现. 而不必集成在 Yosart 中.

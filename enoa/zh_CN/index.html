Introduction

ENOA

enoa ?

enoa 是一个小巧且全面集成的应用框架, 一切都可以轻松配置, 抛却繁杂的各种框架复杂的使用方式, 转为统一实现.

enoa 在实现这个目标的同时坚持着自身的小巧, enoa 从最初开始只是一个简单的工具库的轮子而已, 当然你会说工具库有 common-lang 这样的工具包为什么还要自己造? 是的 common-lang 这样的工具库已经非常完善, 但是随着不断发展很多东西都在改变, 一方面 common-lang 在 API 方面, 命名风格, 使用方式均比较繁杂, 且 common-lang 不断发展多个版本的出现已经出现兼容性的问题, 而不同的框架对其的依赖版本也不禁相同, 在 enoa 的开发过程中就有遇到该问题 (tio-httpserver:commons-lang3:3.7 & rocker:commons-lang3:3.3.2 视类加载顺序, 会导致 tio-httpserver 或 rocker 无法工作) 平白增加开发人员工作.

当然现在 enoa 不仅仅就只是一个工具库了, 与工具库类似, 另一个着眼的东西是 repeater 一个 web 开发框架, 在 java web 方面, Spring 系列可以说是圣经, 非常完善的体系, servlet 事实上的标准实现; 从早些年 java web 界的 SSH 体系, 到 Spring MVC 以及现在的 Spring Boot, Spring Cloud. 不断的在进化, 不过 Spring 系列在开发过程中一直很繁琐, 一大片的 XML 配置, Spring Boot 的出现一定程度上简化了 Spring 原有体系, 减少了 XML 配置文档, 转而投向注解, 也就导致了现在代码里到处充斥着注解, 注解有一个很大的弊端是无法直观的理解, 当工程中出现 20+ 甚至更多种注解, 在阅读方面简直就是灾难, 你无法单单通过注解去理解如何运作, 此外很多框架都会提供自己的注解方案, 导致有大量命名重复的注解, 造成了开发人员在选用上的障碍:

典型的案例

  • Spring Data JPA 以及 Mybatis 混用的情况, 事物注解 Transactional
  • Spring 提供的 Controller 参数注解 Param, Fegin Param, Mybatis Param

Spring 核心 IoC 也成了开发过程中一个严重的问题, 大量的反射导致项目启动需要等待长久的事件, 工程越大启动时间越久. 这是难以忍受的.

至此, 关于 Spring 的问题不再赘述, 不是这里讨论重点, 简单来说不管是 Spring 还是其他一些框架都存在一个弊病 臃肿. 如果你觉得这些都不是问题, 你可以接收并且觉得很便捷, 那么你可以在此处放弃继续阅读, enoa 可能不适合你的胃口 :)

而 repeater 则是完全不同的设计, repeater 和其名字一样, 只是一个中继器, 实际上 repeater 并不能算得上是一个 web 框架, 因为其并没有任何做任何和 web 框架该做的事情, repeater 只是很简单的将不同的 web 服务器/容器 抽象成统一实现, 当下支持的 web 服务器/容器 包括:

  • fastcgi
  • jetty
  • nanohttpd
  • netty
  • tio-httpserver
  • tomcat

这之中有很小的 nanohttpd 服务, 也有功能全面完善的 servlet 容器, 高性能 netty, tio-httpserver. 而 fastcgi 的支持可以做到兼容支持 fastcgi 的 web 服务器接入, 例如 Nginx, lighttpd 等. 但是这并不是全部, 如果你愿意, 任何 web 服务器都可以接入到 repeater. 后面将会详细介绍.

尽管如此, 仍然保持着轻量的体系结构, 简单易懂.

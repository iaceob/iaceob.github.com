Repeater

Repeater

介绍

Repeater 组件是 enoa 生态体系中最重要的一环, enoa 提供的各种包装器, 以及上游的 Web 框架都是通过 Repeater 进行连接的.

简单来说 Repeater 所做的工作是, 创建一个 Web 服务, 并提供接口进行上游工程进行处理.

需要注意的一点是, 虽然是创建 Web 服务, 但是 Web 服务却并不是 Repeater 创建的, 刚看到这个概念可能不能理解.

Repeater 的出现也并不是偶然, 就像本文开头说的 Java web 很多框架都是非常臃肿的, 就只是写个简单的 Hello word 要写一堆配置文档, 数个依赖包, 工程启动要来个十几秒, 打包后的体积几十 MB , 每次等待启动完成的过程中都不禁感叹, 我到底在写什么浩大的工程?

当然后期是完全跳出了这样的环境, 认识了 JFinal , JFinal 可以说是一个及其简单实用的 Servlet 框架, 终于不用再维护一堆配置文档, 工程启动也不必要大量的反射来进行 IoC, 速度也非常快, 很长的一段时间都是在使用 JFinal.

然而, JFinal 也存在一个问题, 因为 JFinal 的发行是将所有功能都放在一起, 也有了臃肿的姿态 (其实体积上没有太多臃肿), 然而在工程开发中, 很多东西都是不必要的, 也因为都放在一起很多功能都有一定的关联. 另外 JFinal 也是基于 Servlet 实现的, 因此也就限定只能运行在 Servlet 容器中.

Repeater 出现的直接原因就是要写一个很简单的 API 接口工程, 完全不想考虑使用 JFinal, JFinal 打包了太多东西, 而这些都不是想要的, 模板 redis 数据库操作, 在这个工程都是不必要的, 但是无法选择, 因此就找到了一个很轻的 Web 服务器 Nanohttpd, 然而在使用的过程中并不愉快, 简单的接口还好, 万一这个工程以后需要扩展, 就会变得很麻烦, 又得重写.

这时候就生出了封转 Nanohttpd 的念头, 起初 Repeater 的目的就只是一个基于 Nanohttpd Web 框架, 然后又想了一下, 这里确实用 Nanohttpd 合适, 但是如果限定使用 Nanohttpd, 其他工程却不合适, 又得使用其他的框架. 因此 Repeater 就变成了现在模样 (这些可以从早期的 Repeater 代码提交中可以观察到, 现在的 Yosart 相关接口就存在与 Repeater 中).

Repeater 一个 Web 服务中继器, 可以接入任何 Web 服务, 并转化为统一实现, 这样可以根据工程类型的不同而选择不同的 Web 服务器而业务代码却不用作出任何更改.

WEB 服务器接入

目前 Repeater 支持的 Web 服务器/容器 包括

  • fastcgi
  • jetty
  • nanohttpd
  • netty
  • tio-httpserver
  • tomcat

其中 nanohttpd 是一个非常轻量级的 Web 服务器, 非常适合嵌入式结构中, 常见与 Android 使用, fastcgi 主要是一些 Web 服务服务器使用, 包括 Nginx, Lighttpd 等, PHP 大多就是通过 fastcgi 接入到 Web 服务器中运行的. jetty 以及 tomcat 是标准的 Servlet 容器. Netty 和 tio-httpserver 其擅长与常连接, 并且提供 Web 服务器功能.

结构图

Repeater 的工作流程就和图中描述的相同, 核心的三个接口是 RepeaterServerFactory , EoxAccessor, RepeaterTranslateFactory 这三个接口的配合使之达到 Web 服务转出的目的.

  • RepeaterServerFactory

    Web 服务器提供者接入接口, 服务提供者要求提供 listen 方法, 返回一个 RepeaterServer 进行实际的服务监听.

  • EoxAccessor

    Web 服务访问器, 所有 Web 服务器接入后, 所有的请求, 都转交给 EoxAccessor 进行处理, Repeater 的请求入口接口.

  • RepeaterTranslateFactory

    Repeater 有自己的 Request 以及 Response, 而不是使用提供者的 Request 和 Response 以做到 Repeater 的通用性, RepeaterTranslateFactory 的作用就是将提供者的 Request 翻译成 Repeater 的 Request, 以及将 Repeater 的 Response 再翻译成提供者识别的 Response 交给提供者进行响应.

理解这些概念之后, 再来看这张图, 首先 Repeater 调用选择的提供者进行 listen, 当服务启动后, 有请求进来, 首先进行的操作就是第一次调用 RepeaterTranslateFactory 将提供者 Request 翻译成 Repeater Request, 翻译完成后, 调用 EoxAccessor 进行请求处理, 处理完成后, 再次调用 RepeaterTranslateFactory 进行 Response 的翻译, 还原成提供者识别的 Response 并返回, 一个生命周期在此结束.

其他接口

RepeaterFactory

RepeaterFactory 是 Repeater 提供对外的以接口, 该接口可以指定 Repeater 启动的一些环境配置.

static RepeaterFactory def() {
  return EnoaRepeaterAccessorImpl::new;
}

EoxAccessor accessor();

default EoLogFactory log() {
  return new JdkLogProvider();
}

default EoxNameRuleFactory rule() {
  return new OriginNameRule();
}

default EoxErrorRenderFactory errorRender() {
  return EoxErrorRenderFactory.def();
}

可以指定 accessor 访问器, log 实现, 命名规则, 以及错误统一渲染.

log() 使用的便是包装器中的 log 组件, rule() 命名规则用于文档上传临时存储时使用的命名规则, errorRender() 错误统一渲染, 用于运行期间发生错误后的渲染页面内容.

以上三个方法都是有默认值的, accessor() 在 Repeater 的启动中必须要指定一个访问器, 如果 Repeater 启动中没有指定, 将使用自带的 EnoaRepeaterAccessorImpl 访问器.

EoxNameRuleFactory

Repeater 对外置口, 有默认的实现: OriginNameRule, 该命名规则使用上传的文档原名, 如果重复在原文档名上增加数字区别重复.

EoxErrorRenderFactory

Repeater 对外置口, 有默认实现: EoxErrorRenderImpl, 仅是简单的输出错误消息

EoxProviderFactory

Repeater 内部接口, 在使用 Repeater 中没有机会接触到, 但是若要开发一个提供者会用到, 该接口是将上面提到的 RepeaterTranslateFactory, EoxNameRuleFactory, EoxErrorRenderFactory 三个接口打包在一起传递给提供者, 供提供者使用.

Request

Repeater 统一 Request 接口, 所有接入的提供者均要求将 Request 翻译成 Repeater 的 Request 接口.

_RepeaterPlusRequest

Request 功能增强接口, 此接口外部无需关注, 只有在开发提供者时会用到, 因为各种不同的 Web 服务器的 Request 功能不尽相同, 在 attribute 以及 context 的支持上, 并不是所有的都有, 而且有些虽然有但是却不完善, 此接口是将提供者 Request 完善的接口.

_RepeaterPlusRequestContext

与 _RepeaterPlusRequest 相同, 不过只是增强 context 的支持

SSL

SSL 的支持在 Repeater 中已存在. 一个关键的配置类是 EoxSSLConfig, 不过因为各服务器的 SSL 启动方式都相对叫复杂, 调试相对较困难, 暂时尚未实际支持.

另外 fastcgi 是无需支持 SSL , 只需要在相应的 Web 服务器配置即可, 例如 Nginx.

文档上传

各 Web 服务器/容器 对文档上传的支持也不禁相同, 有的有, 有的没有, 下面列出目前已支持的服务器对文档上传的支持情况

SERVER SUPPORT DESCRIPTION
FastCGI NO 不支持
Jetty YES Jetty 自身支持文档上传, 需要设置参数 (Repeater 中已设置)
NanoHttpd - NanoHttpd 自身是有文档上传的, 但是存在很多问题
Netty YES
tio-httpserver YES tio 支持文档上传, 不过其实现方式和其他服务器的方式完全不同
Tomcat YES 与 Jetty 类似.

  • NanoHttpd

    NanoHttpd 实际上是支持文档上传的, 不过存在较多问题, 其上传的文档强制按照其内部实现方式并不容易进行提取, 且放置与 http 参数中, 因此 Repeater 中, 未支持 Nanohttp 的默认文档上传. 实际上其官方提供的文档上传案例也不是使用默认方式, 而是通过 common-upload 组件实现.

  • tio-httpserver

    tio-httpserver 支持文档上传, 但是其实现方式与其他服务器的支持完全不同, 其他服务器对于文档上传都是通过 InputStream 或者提供相应的方法进行获取. tio-httpserver 不同的是, 其不提供获取 InputStream, 而是将上传的文档放置与参数中, 该参数值的类型是 UplodFile, 且获取的值是 byte 数组, 因此 Repeater 的做法是直接将该数组保存到文档中去.

注意:

Repeater 的文档上传机制实际上对各提供者做了改造, 通常文档上传到临时文档并不会在接收到请求就马上进行保存, 而是在调用获取文档的方法后再保存到临时文档去.

然而这是不必要的, 在 Repeater 中, 文档上传接收到请求后就会将文档写入到预先配置的临时目录中. 后续通过 Request 获取到的已经是存储后的文档信息.

本身文档上传就应该是 Web 服务器应该做的事情, 而上游的应用服务应该专注与业务逻辑. 且在 Servlet 中也有一些限制, 文档上传的表单, 必须要先获取文档后才能获取到其他参数, 否则会导致无法获取文档的问题, 在 Repeater 的统一结构下, 避免了这个问题, 上游可以随意顺序调用, 因为文档上传已经被 Repeater 处理国.

COS

为了弥补各服务器的文档上传的支持不同, 而引入的第三方上传组件, COS 组件来自于 oreilly cos 是一个高性能的文档上传组件, cos 上传组件是基于 Servlet 实现的, 这里将 cos 组件拿过来剔除了 Servlet 依赖, 且删除了和文档上传无关的类. cos 组件可在源码中 enoa-repeater-cos 获取.

各服务器对 cos 上传组件的支持

SERVER SUPPORT
FastCGI YES
Jetty YES
NanoHttpd YES
Netty YES
tio-httpserver NO
Tomcat YES

  • NanoHttpd

    cos 上传也是 Repeater 中 NanoHttpd 唯一支持上传的方式

  • tio-httpserver

    tio 因其特殊性, 无法支持 cos, cos 上传通过 InputStream 进行写入, tio 若要兼容就需要将 byte 数组转化为 InputStream, 这样完全是浪费.

对于本身已支持文档上传的提供者, 默认走的就是提供者自己的上传方式, 若要切换为 cos 上传, 在 Repeater 启动时提供配置告知即可, 下列表格是相应的服务器告知键.

SERVER KEY VALUE DESCRIPTION
FastCGI 无需提供, FastCGI 文档上传必须使用 cos
Jetty provider.jetty.upload.vendor cos -
NanoHttpd 无需提供, NanoHttpd 文档上传必须使用 cos
Netty provider.netty.upload.vendor cos -
tio-httpserver 不支持
Tomcat provider.tomcat.upload.vendor cos -

如何使用见 案例

Servlet

Repeater 对 Servlet 容器进行了部分改造, 使之与 Repeater 的 Request 机制相同.

  • 在 Servlet 中, Request Body 只能够读取一次, Repeater 中没有此限制.
  • 文档上传表单请求, 必须要先获取文档, 才可获取其他参数, Repeater 中没有此限制.

这些改造是目前已支持的 Servlet 容器, 若有其他的 Servlet 容器接入, 也建议同样对这部分进行改造.

FastCGI

FastCGI 并不是单独启动后直接访问的, 而是借助于支持 FastCGI 的 Web 服务器. 例如 Nginx.

假设 Repeater 使用 FastCGI 提供者启动, 端口是 9100. 那么可以在 Nginx 增加如下配置:

server {
  listen       80;
  server_name  localhost;

  location / {
    root           html;
    fastcgi_pass   localhost:9100;
    include        fastcgi_params;
  }
}

接下来请求 Nginx 的绑定的端口即可, 例如这个配置, 可以直接请求 localhost

案例

Repeater 的使用非常简单, 下面是个最简单的启动 Repeater 代码.

Repeater.createServer(new NanoHTTPDProvider())
      .listen(9100);

这样就启动了一个基于 NanoHttpd 的 Web 服务器.

当然, 这样是没有意义的. 接下来看看高级用法.

EoxConfig config = new EoxConfig.Builder()
  .debug(true)
  .info(true)
  .infoUseLog(false)
  .context("/example")
  .tmp(String.format("%s%stmp", PathKit.getRootPath(), File.separator))
  .other(Kv.by("provider.jetty.upload.vendor", "cos"))
  .build();

RepeaterFactory factory = new RepeaterFactory() {
  @Override
  public EoxAccessor accessor() {
    return new ExampleRepeaterAccessorImpl();
  }

  @Override
  public EoLogFactory log() {
    return new Slf4JLogProvider();
  }
};

Repeater.createServer(new JettyProvider())
  .ssl(false)
  .factory(factory)
  .config(config)
  .listen(9100);

这样就启动了一个 Jetty 的 Web 服务器.

这里面涉及了两个重要的方法: factory, config

  • factory()

    Repeater 启动组件依赖

  • config()

    Repeater 启动配置

factory 要求 RepeaterFactory, 这个之前已经介绍过了. 这里来说说 config.

config 要求提供 EoxConfig, EoxConfig 中有如下配置

  • debug

    是否 debug 模式

  • context

    web 请求 context

  • charset

    编码

  • info

    是否打印 Repeater 信息

  • infoUseLog

    Repeater 相关信息默认使用 System.out.print 打印, 如果这里设置 true, 则会使用 Log 组件打印.

  • soTimeout

    请求超时时间

  • tmp

    文档上传临时存储路径

  • maxUploadSize

    文档上传最大尺寸, 单位 MB

  • ssl

    如果设置为 ssl 启动, 此处是 ssl 的相关配置

  • other

    其他资源配置, 例如 cos 文档上传告知键就放置与此

Repeater 的使用就这样结束了, 更多丰富的功能直接实现 EoxAccessor 接口, 在 factory 中提供即可, 你完全可以基于 Repeater 创建一个自己的 Web 框架, Repeater 已经将不同的 Web 服务器统一输出, 剩下要做的就只是实现一个访问器.

更多案例, 可在源码中 /enoa-example/example-repeater 获取, 并且在 /enoa-test/repeater 中有测试脚本, 测试 Repeater 功能.

Yosart

Yosart

Yosart 是一個基於 Repeater 的 Web 框架, Yosart 的涉及極其精簡, 其核心只是提供基本的 Web 服務的接口, 剩下的交給插件或擴展去實現.

結構圖

還記得 Repeater 的結構圖嗎? Yosart 是 Repeater 的上游高級應用, 實現了 Repeater 提供的 EoxAccessor 接口, 增加了路由, 渲染等功能.

圖中右側便是 Yosart 的結構圖, 接入到 Repeater 體系中.

首先來看看 Yosart 中重要的幾個接口和類

  • OysartAccessor

    Yosart 訪問器, 提供給 Repeater, 用於接收請求

  • OyResource

    Yosart 所有控制器分析後的資源, 包括 YoAction 以及 YoControl

  • YoExt

    Yosart 擴展接入接口

  • YoPlugin

    Yosart 插件加入接口

核心

Yosart 核心部分均在 io.enoa.yosart.kernel 包下, kernel 下的所有包, 以下核心便是 Yosart 所做的事, 拋卻這些核心, 剩下的全部交給 插件 或 擴展 進行處理.

anno

註解包, Yosart 提供的註解, 僅有一個 Action 註解, 該註解用於申明接口 uri 地址, 以及允許的 http method.

data

數據承載實體類

ext

關於 Yosart 的擴展, 需要明白, 所有的擴展均來自於 YoExt, 不過 YoExt 並不是被直接實現的, 而是通過具體不同的接口實現, 例如 渲染擴展 (YmRenderExt) 路由擴展 (YoRouterExt) 等, 詳細見 擴展

http

Yosart 基於 Repeater , 但是 Repeater 的 Request 僅僅是簡單的 http 請求信息, 但是對於 Web 框架而言這些是不夠的, Yosart 支持 Restful 風格的 API, 這些 Repeater 並沒有做, 且 Repeater 也未提供將請求參數轉換爲實體的方法, http 包下是 Yosart 對 Repeater 的 Request 的高級封裝.

mount

mount 中只有一個 OyMountd 類, 在正常的業務邏輯開發中是用不到的. Yosart 提供一個信息輸出的頁面 (後續會說道). 而 OyMountd 是將其他組件的信息掛載到 Yosart 中.

render

所有渲染擴展最終渲染的返回接口 YoRender

resources

Action 資源包, YoAction , YoRouter 以及 YoControl 均定義再此, 也包含所有 Action 資源的分析, 最終處理成 OyResource

Resp

Yosart 除了核心包之外, 還提供了 Response 構造器, io.enoa.yosart.resp 包下, 存在兩種構造器

  • Resp
  • Renderer

區別是, Renderer 所構造的內容都是通過渲染擴展進行渲染的內容, 而 Resp 可以構造渲染內容之外, 還支持 Response 相關值的設定, 包括 Header Cookie 等.

Restful

Yosart 支持 Restful 風格接口, 接口 uri 定義遵循以 : 表明參數名, 例如以下案例

GET  /api/user/:id
PUT  /api/user/:id

即可通過 Request 中 variable("id") 的方式獲取.

插件 & 擴展

關於插件和擴展的區別, 早些年在 firefox 貼吧中有一句話描述的很好 扩展负责扩开来, 插件负责插进去.

插件

Yosart 插件, 只需要實現 YoPlugin 接口即可, 啓動時, 加入到 Yosart 中.

擴展

Yosart 擴展目前支持的有

  • YmAssetsExt

    靜態資源擴展

  • YmBeforeActionExt

    Action 前置擴展, 用於在執行 Action 前執行的操作

  • YmBootHookExt

    啓動鉤子, 會在 Yosart 啓動之前調用的擴展

  • YmExceptionExt

    用於在調用 Action 出現異常的情況下執行的擴展

  • YmRenderExt

    渲染擴展

  • YmRouterExt

    路由擴展

需要注意的是, 這裏理出的擴展並非都是 Yosart 直接支持, Yosart 直接支持的有 YmAssetsExt, YmRenderExt, YmBootHookExt 以及 YmRouterExt.

YmBeforeActionExt 和 YmExceptionExt 視選擇的 YmRouterExt 是否支援, Yosart 無法控制, 如果選擇的路由擴展不支持 YmBeforeActionExt 和 YmExceptionExt 則無法使用.

自定義擴展需要注意 weight 方法, 該方法的含義是此擴展的權重, 權重將影響擴展的執行順序, 可取值範圍 1-9 之間浮點數, 0-1 以及 9-10 爲 Yosart 保留權重. 默認會隨機生成, 無需關注.

使用

Yosart 自帶 EyRouterExt 路由擴展, 因此下面描述的使用方式均是 EyRouterExt 所支持的.

一個簡單的 Yosart 使用案例.

Yosart.createServer()
  .handle(Method.GET, "/", (req) -> Resp.with(req).render("yosart").end())
  .listen();

Yosart 默認搭載的 Web 服務器是 NanoHttpd, 默認監聽端口 9102, 只需在工程中引用 Repeater 以及 NanoHttpd 即可.

是的, 這個很簡單, 接下來看一個很完善的使用案例

Yosart.createServer()
  .config(this.config())
  .repeater(new YasartRepeaterFactory())
  .provider(new NanoHTTPDProvider())
  .log(new Slf4JLogProvider())
  .plugin(new JsonPlugin(new EoJsonProvider()))
  .plugin(this.activeRecordPlugin())
  .plugin(this.beetlSQLPlugin())
  .plugin(this.mybatisPlugin())
  .plugin(new ConfPlugin("https://github.com/mstine/config-repo.git"))
  .ext(new JsonRenderExt())
  .ext(new TemplateRenderExt(TEMPLATE.ENJOY.factory, TEMPLATE.ENJOY.basePath, TEMPLATE.ENJOY.suffix, new HtmlCompressorProvider()))
  .ext(new BeaExt())
  .ext(new YemgrExt())
  .context("/example")
  .assets("/assets", Paths.get(EnvKit.get("java.io.tmpdir")), true)
  .handle(Method.GET, "/", (req) -> Resp.with(req).attr("req", req.data()).renderTemplate("info").end())
  .handle(YosartControl.class)
  .handle(new WorkRouter())
  .handle("/yosart", (req) -> null)
  .handle("/party", PartyControl.class)
  .handle("/corpse", new ZombieRouter())
  .handle("/download", (req) -> Resp.with(req).renderFile(Paths.get(String.format("%s/download/file.txt", PathKit.getRootPath()))).end())
  .handle("/db", DbControl.class)
  .handle("/actionanno", new YoAction() {
    @Before({ActionAnnoValid.class, PartyGoHandler.class})
    @Override
    public Response handle(YoRequest request) {
      return Resp.with(request).render("action anno").end();
    }
  })
  .listen(9102);

案例代碼可通過源碼 /enoa-example/example-yosart 獲取

這個案例中涉及到了很多, 插件 擴展, 這些不屬於 Yosart 要介紹的部分, 去查閱相關插件或擴展的文檔. 這裏主要介紹與 Yosart 實現有關的部分.

在說路由之前, 先看以下 Yosart 啓動接口, 該接口是 Yosart 最中的地方, 所有插件, 擴展 路由都是在這裏註冊. 下面列出 Yosart 所有方法

  • provider

    選擇 Web 服務器提供者, 視需要選擇一個提供者.

  • repeater

    Repeater 配置項, EoxAccessor 訪問器配置就在此處, Yosart 提供了一個默認的配置 YasartRepeaterFactory, 通常這個是不需要進行配置的, 如果自定義了其中了 accessor 那麼也就表示着放棄了 Yosart 提供的請求解決方案, 那麼就去看相應的 accessor 是如何工作的.

  • log

    Yosart 日誌提供者選擇

  • plugin

    添加插件, 插件添加後會馬上執行.

  • ext

    添加擴展, 擴展在服務監聽前一起執行

  • ssl

    開啓 ssl 服務

  • handle

    註冊路由

  • name

    應用名稱

  • context

    Http context

  • config

    Yosart 配置信息

  • assets

    靜態資源配置

  • listen

    服務監聽

這裏主要說下 handle 以及 assets 方法

handle 路由註冊, 供提供了以下幾種方式

Yosart handle(Method method, String uri, YoAction action);

Yosart handle(String uri, YoAction action);

Yosart handle(String uri, Class<? extends YaControl> control);

Yosart handle(String uri, YoRouter router);

Yosart handle(Class<? extends YaControl> control);

Yosart handle(YoRouter router);

YoAction

YoAction 提供兩種註冊方式, 限定 Method 和不限定, 當然這個具體是什麼含義視路由擴展如何定義, 默認的路由擴展中, 沒有 Method 則表明允許任何 Method 請求, 這個概念在 Control 中也適用.

YoAction 定義方式

new YoAction() {
  @Override
  public Response handle(YoRequest request) {
    return Resp.with(request).render("action").end();
  }
}

也可以使用 Java 8 提供的 lambda 表達式方式

(req) -> Resp.with(req).render("action").end()

YoControl

YoControl 註冊, 需要將註冊的控制器繼承 YoControl/YeControl/ YaControl 類.

之所以提供三種不同 Control 是爲了不同的路由實現考慮, 這三種不同的 Control 的區別

CONTROL DESCRIPTION
YaControl 直接實現自 YaResource 接口, 內部存在 Request 以及 Resp
YeControl 繼承自 YaControl, 提供將 Request 相同方法而不需要獲取 Request 後再獲取參數信息
YoControl 繼承自 YoControl, 提供 render 系列方法

應用場景不同, 例如默認的路由實現, 因爲不支持控制器方法參數解析以及方法的返回值解析, 那麼使用 YoControl 方法最合適, render 系列方法則完成了所有渲染功能, 而不用關心方法的參數以及返回值; 另一個路由 yosart-ext-router-anost 則支持方法參數的解析以及方法返回解析, 那麼 render 以及 request 的方法已經沒什麼必要, 完全可以通過方法參數獲取 request 參數, 因此可以選擇繼承 YaControl, 避免開發中 control 的方法污染. 選擇哪個取決於所使用的路由擴展, 當然不選擇全部用 YoControl 也是可行的.

使用方式

public class IndexControl extends YoControl {
  public void index() {
    super.render("index");
  }

  @Action(method = Method.GET, value = "/action_test")
  public void actionTest() {
    super.render("action test");
  }
}

需要注意, 默認的路由擴展不支持 Control 方法參數解析, 另也無法解析方法的返回值, 當然就算寫上也不會報錯, 將會收到警告日誌, 如果需要支持請選擇其他路由實現擴展. (官方攜帶的 yosart-ext-router-anost 將會支持. 目前尚未施工完畢.)

YoRouter

還有一種註冊方式是 YoRouter 路由註冊, 該註冊方式區別與 YoAction 以及 YoControl, Yosart 會在啓動時將 YoRouter 分析成控制器或 Action. 關於 YoRouter 可以看下使用方式

public class WorkRouter extends YoRouter {
  @Override
  public String context() {
    return "/work";
  }

  @Override
  protected void register() {
    super.add(IndexControl.class)
      .add("/holiday", HolidayControl.class)
      .add("/working_day", WorkingDayControl.class)
      .add("/report", new ReportRouter())
      .add("/lazy", (req) -> null);
  }
}

當繼承了 YoRouter 之後, 實現兩個方法, context 以及 register, context 的意思是該路由所在的 context, 這個 context 需要瞭解的是前後文的 context.

例如

.handle("/day", new WorkRouter())

那麼註冊的路由就是

/day/work/holiday
/day/work/working_day
...

如果不需要該 context, 直接留空即可.

register 方式用於註冊該路由下的控制器, 同樣的也是支持 YoAction, YoControl 以及 YoRouter

assets

另外一個要說的是 assets 方法, 該方法的作用是創建一個靜態資源請求的支持.

Yosart assets(String uri, Path path);

Yosart assets(String uri, Path path, boolean showTree);

uri 爲靜態資源的前綴, path 爲靜態資源路徑, showTree, 表示在請求的路徑是目錄時是否顯示文件列表. 靜態資源是走擴展實現的, 默認實現爲 EyAssetsExt, 如果想要其他的實現, 實現 YmAssetsExt 接口加入到 yosart 啓動中即可.

YoRequest

YoRequest 在 Repeater 的基礎上新增 Restful 風格接口 url 參數解析, 以及參數轉換 bean 方法.

  • variableNames

    獲取所有 uri 參數名

  • variable

    獲取參數值

  • bean

    將 Request 參數轉化爲實體類

其他

session

可能在使用過程中你會發現, Yosart 並未提供 session 的操作功能, 可能無法理解, 很多框架都可以進行 session 操作, 爲什麼 Yosart 不支持?

首先, 請理解 session 的本質, session 實際上是通過 cookie 去實現的, 創建一個 session 實際上是推送一個 cookie 給前端, 每次請求都會將 cookie 帶回來, 服務端通過 cookie 去查找之前設置的 session 信息.

那麼, 既然 Yosart 支持 cookie 獲取, session 完全可以通過擴展的方式實現. 而不必集成在 Yosart 中.

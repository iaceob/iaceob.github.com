Introduction

ENOA

正體中文文檔請注意, 本文雖採用正體中文撰寫, 但因存在語言環境差異, 文中很多用詞習慣可能並非完全屬於當地的用語.

enoa ?

enoa 是一個小巧且全面整合的應用框架, 一切都可以輕鬆配置, 拋卻繁雜的各種框架複雜的使用方式, 轉爲統一實現.

enoa 在實現這個目標的同時堅持着自身的小巧, enoa 從最初開始只是一個簡單的工具庫的輪子而已, 當然你會說工具庫有 common-lang 這樣的工具包爲什麼還要自己造? 是的 common-lang 這樣的工具庫已經非常完善, 但是隨着不斷發展很多東西都在改變, 一方面 common-lang 在 API 方面, 命名風格, 使用方式均比較繁雜, 且 common-lang 不斷發展多個版本的出現已經出現兼容性的問題, 而不同的框架對其的依賴版本也不禁相同, 在 enoa 的開發過程中就有遇到該問題 (tio-httpserver:commons-lang3:3.7 & rocker:commons-lang3:3.3.2 視類加載順序, 會導致 tio-httpserver 或 rocker 無法工作) 平白增加開發人員工作.

當然現在 enoa 不僅僅就只是一個工具庫了, 與工具庫類似, 另一個着眼的東西是 repeater 一個 web 開發框架, 在 java web 方面, Spring 系列可以說是聖經, 非常完善的體系, servlet 事實上的標準實現; 從早些年 java web 界的 SSH 體系, 到 Spring MVC 以及現在的 Spring Boot, Spring Cloud. 不斷的在進化, 不過 Spring 系列在開發過程中一直很繁瑣, 一大片的 XML 配置, Spring Boot 的出現一定程度上簡化了 Spring 原有體系, 減少了 XML 配置文件, 轉而投向註解, 也就導致了現在代碼裏到處充斥着註解, 註解有一個很大的弊端是無法直觀的理解, 當工程中出現 20+ 甚至更多種註解, 在閱讀方面簡直就是災難, 你無法單單通過註解去理解如何運作, 此外很多框架都會提供自己的註解方案, 導致有大量命名重複的註解, 造成了開發人員在選用上的障礙:

典型的案例

  • Spring Data JPA 以及 Mybatis 混用的情況, 事物註解 Transactional
  • Spring 提供的 Controller 參數註解 Param, Fegin Param, Mybatis Param

Spring 核心 IoC 也成了開發過程中一個嚴重的問題, 大量的反射導致項目啓動需要等待長久的事件, 工程越大啓動時間越久. 這是難以忍受的.

至此, 關於 Spring 的問題不再贅述, 不是這裏討論重點, 簡單來說不管是 Spring 還是其他一些框架都存在一個弊病 臃腫. 如果你覺得這些都不是問題, 你可以接收並且覺得很便捷, 那麼你可以在此處放棄繼續閱讀, enoa 可能不適合你的胃口 :)

而 repeater 則是完全不同的設計, repeater 和其名字一樣, 只是一個中繼器, 實際上 repeater 並不能算得上是一個 web 框架, 因爲其並沒有任何做任何和 web 框架該做的事情, repeater 只是很簡單的將不同的 web 服務器/容器 抽象成統一實現, 當下支持的 web 服務器/容器 包括:

  • fastcgi
  • jetty
  • nanohttpd
  • netty
  • tio-httpserver
  • tomcat

這之中有很小的 nanohttpd 服務, 也有功能全面完善的 servlet 容器, 高性能 netty, tio-httpserver. 而 fastcgi 的支持可以做到兼容支持 fastcgi 的 web 服務器接入, 例如 Nginx, lighttpd 等. 但是這並不是全部, 如果你願意, 任何 web 服務器都可以接入到 repeater. 後面將會詳細介紹.

儘管如此, 仍然保持着輕量的體系結構, 簡單易懂.

其他信息

JFinal

在這裏特別說以下與 JFinal 的關係, 包括從源碼中以及使用方式上, 有很多都和 JFinal 相似, 特別是在 toolkit, json 這些基礎模塊中, 很多都是從 JFinal 中搬運過來的, 實際上 JFinal 也是一個很簡單的框架, 不過隨着發展變得東西越來越多, 但是工程中這些是用不到的, 在我個人的想像中這些應該都是可選項, 應該在我想要的時候出現, 而不是打包一個全家桶.

但是, 要特別聲明的是, 雖然很多工具類都是從 JFinal 中搬運過來, 甚至 Yosart 的默認路由實現都和 JFinal 極其相似, 實際上內部實現原理已經和 JFinal 有很大差別, 就只是使用方式大體相同而已, 之所以 Yosart 默認路由器會這麼做, 最主要的考慮是 JFinal 的一些接口使用方式都相對而言非常簡單.

只要你想, Yosart 可以長成任何模樣, 甚至是 Spring 的方式, 因爲所有都是可配置實現的.

OkHttp

Enoa 的源碼中有很多數據類, 像是模板的配置類, DB 配置類 等等, 創建方式均借鑑自 OkHttp 的類創建方式, 使用 Builder 創建, 無 setter 方法. 避免了數據遭到篡改問題.

且 Repeater 中, 部分代碼來自於 OkHttp, 例如 Cookie

命名

在 Enoa 的整個源碼中充斥着大量的以 _ 開頭命名的類, 以及無修飾符類, 特別是在 Yosart 源碼中, 很多以 Yo Ya Ye Ym Oy Ay 開頭的類.

這只是一個編碼潔癖而已, 或者說 Enoa 的所有 接口, 代碼 都是被精心設計的. 主要原因有以下幾點:

首先以 _ 開頭的類, 先列出以下幾個

  • _FreemarkerEngine
  • _FreemarkerTemplate
  • _RepeaterPlusRequest
  • _RepeaterPlusRequestContext
  • _Json

_FreemarkerEngine, _FreemarkerTemplate 這兩個類都是第三方的具體實現類, 別且無修飾符

_RepeaterPlusRequest, _RepeaterPlusRequestContext 這兩個接口是 Repeater 接入第三方 Web 服務器時, 對第三方 Request 的增加接口, 修飾符爲 public

_Json EnoaJson 的父類, 無修飾符

意圖很明顯

第一個, 因爲是第三方的實現類, 與 enoa 無太大關係, 用以區別, 同類型的很多, 例如 _NanoApp _Jetty 等等.

第二個, 雖然是也是提供給第三方實現用, 不過是 enoa 體系中的接口, 而且修飾符是 public, 而在正常開發中, 這些接口是不可能被用到的, 只在進行第三方接入的時候才會用到, 不想在應用開發中, IDE 打出 Repeater 時會將這些接口也列出來, 造成選擇障礙.

第三個, enoa 體系中的接口, 之所以提出來, 因爲 EnoaJson 是一個抽象類, 有自己的代碼要實現, 如果接口放在抽象類中會很亂, 因此就提出來.

Yosart 中的命名也是相同, 一句話總結就是: Yo 開頭的類全部都是應用開發時會用到的類, Ya 開頭的都是父類, Oy Ay 這些開頭都是開發中幾乎沒有機會用到的類.

此外 Yosart 也沒有佔用很常用的名稱, 包括 Control Action, 這些交給開發者自己去使用.

風格

鏈式調用

Yosart 的代碼風格偏向與鏈式調用, 從 Repeater 以及 Yosart 中就可以看出, 直接一個鏈式全部完成.

getter&setter

無 getter setter 方法, enoa 幾乎所有的代碼都沒有 getter setter 方法, 而是直接使用與變量名同名的方法.

String val = request.attr("key"); // 獲取 attribute

request.attr("key0", "val").attr("key1", "val"); // 設置 attribute

但是這樣也帶來了一定的風險, 太多工具, 特別是 json 工具類都是通過 getter setter 方法獲取或者寫入的, 在 Request 中就遇到了這個問題, 因此 Yosart 中特別增加了一個 data 方法來彌補.

無配置文件

無配置文件, enoa 的所有工程都不需要配置文件, 一切都用代碼去完成, 忘掉配置文件, Java 的笨重很大程度上是早期的一些框架帶的風氣, Spring Struts Hibernate 之流, xml 配置寫到懷疑人生, 還很難以理解, 文檔出的一套一套.

少註解

目前, 整個 enoa 體系中, 受到直接支持的註解僅有 Yosart 中的 Action 註解, 用於路由解析. 其他的都交給擴展去做. 之所以這麼做, 是因爲不希望看到 Yosart 會出現向下面類似的代碼:

@ApiOperation(value="Get user list", notes="Get user list", response = Message.class)
@ApiImplicitParams({
  @ApiImplicitParam(name = "page", value = "Page No.", required = false, dataType = "int", paramType = "query", defaultValue = "0"),
  @ApiImplicitParam(name = "size", value = "Page size", required = false, dataType = "int", paramType = "query", defaultValue = "20"),
  @ApiImplicitParam(name = "sort", value = "Sort rule", required = false, dataType = "String", paramType = "query", defaultValue = "id,desc")
})
@ApiResponses(value = {
  @ApiResponse(code = 0, message = "Success！"),
  @ApiResponse(code = 1, message = "Fail！")
})
@RequestMapping(value="/list", method = RequestMethod.GET)
public ResponseEntity<?> list(@RequestBody UserSearch userSearch, 
                              PagedResourcesAssembler<User> assember, Pageable pageable){
  logger.info("API start > user list. paras：{}", memberSearch);
  Page<User> page = userService.list(userSearch, pageable);
  logger.info("API end > user list. returned：{}", page);
  return new ResponseEntity<>(new Message(Result.SUCCESS, assember.toResource(page)), HttpStatus.OK);
}

要多噁心就有多噁心的代碼, 整個這麼長的代碼有用的就一行, 註解比正常代碼還多, 儘管是因爲有 swigger 的原因 (enoa 的文檔生成以及繼承測試後續也會添加).

來看看實際有用的代碼

@RequestMapping(value="/list", method = RequestMethod.GET)
public ResponseEntity<?> list(@RequestBody UserSearch userSearch, 
                              PagedResourcesAssembler<User> assember, Pageable pageable){
  Page<User> page = userService.list(userSearch, pageable);
  return new ResponseEntity<>(new Message(Result.SUCCESS, assember.toResource(page)), HttpStatus.OK);
}

這才是 enoa 想到達到的效果, 一大堆註解不是 enoa 想要的.

註解也是那些從前一大堆 xml 的框架覺得確實有點麻煩而找到的替代的方案, 然而無論怎麼改都逃脫不了難看的宿命, 他們弄的太複雜了, 很簡單的事情.

最後

如果你覺得簡單的 enoa 並不好, 沒有關係, 在 Yosart 方面, 完全可以根據路由擴展實現你想要的, 甚至直接基於 Repeater 來實現一個. 當然 也是可以繼續去用你習慣用的.

enoa 會繼續簡單下去, 並且不斷的完善.

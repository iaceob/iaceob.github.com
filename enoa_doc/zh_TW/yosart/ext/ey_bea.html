拦截器

攔截器擴展

場景

控制器前置擴展是實現自 YmBeforeActionExt 擴展, 在默認路由中, 會在找到請求資源後 首先調用 BeaExt 接下來在進入到相應的資源.

BeaExt 的核心接口是 io.enoa.ext.bea.beaction.Interceptor` 該接是前置的執行接口, 所有的前置類均來自於此接口.

Interceptor

一個 Interceptor 使用案例

public class GoPartyInterceptor implements Interceptor {
  @Override
  public void intercept(Invocation inv) {
    YoRequest req = inv.req();
    Resp resp = inv.resp();
    resp.renderHtml("Interceptor test");
    inv.next();
  }
}

Interceptor 需要注意的一點是, 當實現了 Interceptor 接口, 在 intecept 方法中, 需要調用 inv.next() 才可進入到後續的 Interceptor, 如果不調用, 則會在此直接返回.

靈活運用 Interceptor 可以做很多事情, 例如可以使用 Interceptor 進行權限認證, 如果權限認證沒過, 不繼續向後走, 直接返回消息.

Handler

Handler 接口集成自 Interceptor, 與 Interceptor 的區別是, Handler 不需要調用 next 方法, 該方法被 Handler 進行了封裝, 實現 handle 方法即可.

public class PartyGoHandler implements Handler {
  @Override
  public void handle(YoRequest req, Resp resp) {
    resp.header(new Header("key", "value"));
  }
}

Handler 的應用也很廣泛, 例如完全可以利用 Handler 設計一個 session 管理器.

Validator

Validator 實現自 Interceptor, 功能是對請求參數進行驗證, 如果未通過驗證則直接返回消息, 此功能是基於 Interceptor 的封裝, 用 Interceptor 接口也同樣可實現此功能.

public class PartyGoValid extends Validator {

  @Override
  protected boolean shortCircuit() {
    return true;
  }

  @Override
  protected void valid(YoRequest req) {
    String valid = req.para("valid", "pass");
    if (!"pass".equals(valid))
      super.error("msg.valid.pass", "Can not pass.");
  }

  @Override
  protected void handle(Kv attr, Resp resp) {
    resp.renderHtml(JsonKit.toJson(attr));
  }

}

Validator 主要有三個方法valid , error 以及handle, valid 方法用於進行參數驗證, 如果驗證失敗可以調用error 方法, 傳遞錯誤消息, 當鑑定有錯誤後會進入到handle方法, 進行信息渲染.

shortCircuit 短路配置, 默認為 true, 在 valid 方法驗證時, 是否遇到第一個 error 調用就跳出.

Before

Before 註解是攔截器使用的載體, 在方法的簽名上使用 Before 註解即可使用攔截器.

Control

  @Before({GoPartyInterceptor.class, PartyGoValid.class, PartyGoHandler.class})
  public void go() {
    super.render("go");
  }

Action

new YoAction() {
  @Before({ActionAnnoValid.class, PartyGoHandler.class})
  @Override
  public Response handle(YoRequest request) {
    return Resp.with(request).render("action anno").end();
  }
}

Clear

Clear 註解, 用於清除攔截器, 最常見的場景是, 賬戶登入設定如果使用攔截器實現, 在登入/註冊界面是不需要進行攔截的, 但是其他頁面都需要, 這時可將攔截器設置為全局攔截器, 而登入註冊則使用Clear 清除該攔截器.

@Clear(LoginInterceptor.class)
public void login() {
  super.render("login");
}

使用

Global

全局攔擊器, 如果一個攔截器被設定為全局為全局攔截器, 那麼每個請求都會進入到該攔截器, 全局攔截器的註冊稍有不同, 並非通過註解使用, 而是在創建BeaExt時指定的.

new BeaExt().global(new GlobalHandler())

Control

Control 類型攔截器, 需要放置與 Control 方法簽名上.

Action

Action 同樣支持該攔截器, 需要放置於 handle 方法簽名上.

細膩化控制

相對而言, 只提供Global, Control, Action 三種攔截方式並不全面, 而且Before 以及Clear 註解限制只能在方法簽名, 不可在類的簽名上. 因此無法做到細膩化的配置, 例如只給賬戶模塊增加不同的攔截器, 等等.

後續將會支持該功能.

POM

<dependency>
  <groupId>io.enoa</groupId>
  <artifactId>yosart-ext-bea-beaction</artifactId>
</dependency>

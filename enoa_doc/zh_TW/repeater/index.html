Repeater

Repeater

介紹

Repeater 組件是 enoa 生態體系中最重要的一環, enoa 提供的各種包裝器, 以及上游的 Web 框架都是通過 Repeater 進行連接的.

簡單來說 Repeater 所做的工作是, 創建一個 Web 服務, 並提供接口進行上游工程進行處理.

需要注意的一點是, 雖然是創建 Web 服務, 但是 Web 服務卻並不是 Repeater 創建的, 剛看到這個概念可能不能理解.

Repeater 的出現也並不是偶然, 就像本文開頭說的 Java web 很多框架都是非常臃腫的, 就只是寫個簡單的 Hello word 要寫一堆配置文件, 數個依賴包, 工程啓動要來個十幾秒, 打包後的體積幾十 MB , 每次等待啓動完成的過程中都不禁感嘆, 我到底在寫什麼浩大的工程?

當然後期是完全跳出了這樣的環境, 認識了 JFinal , JFinal 可以說是一個及其簡單實用的 Servlet 框架, 終於不用再維護一堆配置文件, 工程啓動也不必要大量的反射來進行 IoC, 速度也非常快, 很長的一段時間都是在使用 JFinal.

然而, JFinal 也存在一個問題, 因爲 JFinal 的發行是將所有功能都放在一起, 也有了臃腫的姿態 (其實體積上沒有太多臃腫), 然而在工程開發中, 很多東西都是不必要的, 也因爲都放在一起很多功能都有一定的關聯. 另外 JFinal 也是基於 Servlet 實現的, 因此也就限定只能運行在 Servlet 容器中.

Repeater 出現的直接原因就是要寫一個很簡單的 API 接口工程, 完全不想考慮使用 JFinal, JFinal 打包了太多東西, 而這些都不是想要的, 模板 redis 資料庫操作, 在這個工程都是不必要的, 但是無法選擇, 因此就找到了一個很輕的 Web 服務器 Nanohttpd, 然而在使用的過程中並不愉快, 簡單的接口還好, 萬一這個工程以後需要擴展, 就會變得很麻煩, 又得重寫.

這時候就生出了封轉 Nanohttpd 的念頭, 起初 Repeater 的目的就只是一個基於 Nanohttpd Web 框架, 然後又想了一下, 這裏確實用 Nanohttpd 合適, 但是如果限定使用 Nanohttpd, 其他工程卻不合適, 又得使用其他的框架. 因此 Repeater 就變成了現在模樣 (這些可以從早期的 Repeater 代碼提交中可以觀察到, 現在的 Yosart 相關接口就存在與 Repeater 中).

Repeater 一個 Web 服務中繼器, 可以接入任何 Web 服務, 並轉化爲統一實現, 這樣可以根據工程類型的不同而選擇不同的 Web 服務器而業務代碼卻不用作出任何更改.

WEB 服務器接入

目前 Repeater 支持的 Web 服務器/容器 包括

  • fastcgi
  • jetty
  • nanohttpd
  • netty
  • tio-httpserver
  • tomcat

其中 nanohttpd 是一個非常輕量級的 Web 服務器, 非常適合嵌入式結構中, 常見與 Android 使用, fastcgi 主要是一些 Web 服務服務器使用, 包括 Nginx, Lighttpd 等, PHP 大多就是通過 fastcgi 接入到 Web 服務器中運行的. jetty 以及 tomcat 是標準的 Servlet 容器. Netty 和 tio-httpserver 其擅長與常連接, 並且提供 Web 服務器功能.

結構圖

Repeater 的工作流程就和圖中描述的相同, 核心的三個接口是 RepeaterServerFactory , EoxAccessor, RepeaterTranslateFactory 這三個接口的配合使之達到 Web 服務轉出的目的.

  • RepeaterServerFactory

    Web 服務器提供者接入接口, 服務提供者要求提供 listen 方法, 返回一個 RepeaterServer 進行實際的服務監聽.

  • EoxAccessor

    Web 服務訪問器, 所有 Web 服務器接入後, 所有的請求, 都轉交給 EoxAccessor 進行處理, Repeater 的請求入口接口.

  • RepeaterTranslateFactory

    Repeater 有自己的 Request 以及 Response, 而不是使用提供者的 Request 和 Response 以做到 Repeater 的通用性, RepeaterTranslateFactory 的作用就是將提供者的 Request 翻譯成 Repeater 的 Request, 以及將 Repeater 的 Response 再翻譯成提供者識別的 Response 交給提供者進行響應.

理解這些概念之後, 再來看這張圖, 首先 Repeater 調用選擇的提供者進行 listen, 當服務啓動後, 有請求進來, 首先進行的操作就是第一次調用 RepeaterTranslateFactory 將提供者 Request 翻譯成 Repeater Request, 翻譯完成後, 調用 EoxAccessor 進行請求處理, 處理完成後, 再次調用 RepeaterTranslateFactory 進行 Response 的翻譯, 還原成提供者識別的 Response 並返回, 一個生命週期在此結束.

其他接口

RepeaterFactory

RepeaterFactory 是 Repeater 提供對外的以接口, 該接口可以指定 Repeater 啓動的一些環境配置.

static RepeaterFactory def() {
  return EnoaRepeaterAccessorImpl::new;
}

EoxAccessor accessor();

default EoLogFactory log() {
  return new JdkLogProvider();
}

default EoxNameRuleFactory rule() {
  return new OriginNameRule();
}

default EoxErrorRenderFactory errorRender() {
  return EoxErrorRenderFactory.def();
}

可以指定 accessor 訪問器, log 實現, 命名規則, 以及錯誤統一渲染.

log() 使用的便是包裝器中的 log 組件, rule() 命名規則用於文件上傳臨時存儲時使用的命名規則, errorRender() 錯誤統一渲染, 用於執行期間發生錯誤後的渲染頁面內容.

以上三個方法都是有默認值的, accessor() 在 Repeater 的啓動中必須要指定一個訪問器, 如果 Repeater 啓動中沒有指定, 將使用自帶的 EnoaRepeaterAccessorImpl 訪問器.

EoxNameRuleFactory

Repeater 對外接口, 有默認的實現: OriginNameRule, 該命名規則使用上傳的文件原名, 如果重複在原文件名上增加數字區別重複.

EoxErrorRenderFactory

Repeater 對外接口, 有默認實現: EoxErrorRenderImpl, 僅是簡單的輸出錯誤消息

EoxProviderFactory

Repeater 內部接口, 在使用 Repeater 中沒有機會接觸到, 但是若要開發一個提供者會用到, 該接口是將上面提到的 RepeaterTranslateFactory, EoxNameRuleFactory, EoxErrorRenderFactory 三個接口打包在一起傳遞給提供者, 供提供者使用.

Request

Repeater 統一 Request 接口, 所有接入的提供者均要求將 Request 翻譯成 Repeater 的 Request 接口.

_RepeaterPlusRequest

Request 功能增強接口, 此接口外部無需關注, 只有在開發提供者時會用到, 因爲各種不同的 Web 服務器的 Request 功能不盡相同, 在 attribute 以及 context 的支持上, 並不是所有的都有, 而且有些雖然有但是卻不完善, 此接口是將提供者 Request 完善的接口.

_RepeaterPlusRequestContext

與 _RepeaterPlusRequest 相同, 不過只是增強 context 的支持

SSL

SSL 的支持在 Repeater 中已存在. 一個關鍵的配置類是 EoxSSLConfig, 不過因爲各服務器的 SSL 啓動方式都相對叫複雜, 調試相對較困難, 暫時尚未實際支持.

另外 fastcgi 是無需支持 SSL , 只需要在相應的 Web 服務器配置即可, 例如 Nginx.

文件上傳

各 Web 服務器/容器 對文件上傳的支持也不禁相同, 有的有, 有的沒有, 下面列出目前已支持的服務器對文件上傳的支持情況

SERVER SUPPORT DESCRIPTION
FastCGI NO 不支持
Jetty YES Jetty 自身支持文件上傳, 需要設定參數 (Repeater 中已設定)
NanoHttpd - NanoHttpd 自身是有文件上傳的, 但是存在很多問題
Netty YES
tio-httpserver YES tio 支持文件上傳, 不過其實現方式和其他服務器的方式完全不同
Tomcat YES 與 Jetty 類似.

  • NanoHttpd

    NanoHttpd 實際上是支持文件上傳的, 不過存在較多問題, 其上傳的文件強制按照其內部實現方式並不容易進行提取, 且放置與 http 參數中, 因此 Repeater 中, 未支持 Nanohttp 的默認文件上傳. 實際上其官方提供的文件上傳案例也不是使用默認方式, 而是通過 common-upload 組件實現.

  • tio-httpserver

    tio-httpserver 支持文件上傳, 但是其實現方式與其他服務器的支持完全不同, 其他服務器對於文件上傳都是通過 InputStream 或者提供相應的方法進行獲取. tio-httpserver 不同的是, 其不提供獲取 InputStream, 而是將上傳的文件放置與參數中, 該參數值的類型是 UplodFile, 且獲取的值是 byte 數組, 因此 Repeater 的做法是直接將該數組保存到文件中去.

注意:

Repeater 的文件上傳機制實際上對各提供者做了改造, 通常文件上傳到臨時文件並不會在接收到請求就馬上進行儲存, 而是在調用獲取文件的方法後再儲存到臨時文件去.

然而這是不必要的, 在 Repeater 中, 文件上傳接收到請求後就會將文件寫入到預先配置的臨時目錄中. 後續通過 Request 獲取到的已經是存儲後的文件信息.

本身文件上傳就應該是 Web 服務器應該做的事情, 而上游的應用服務應該專注與業務邏輯. 且在 Servlet 中也有一些限制, 文件上傳的表單, 必須要先獲取文件後才能獲取到其他參數, 否則會導致無法獲取文件的問題, 在 Repeater 的統一結構下, 避免了這個問題, 上游可以隨意順序調用, 因爲文件上傳已經被 Repeater 處理國.

COS

爲了彌補各服務器的文件上傳的支持不同, 而引入的第三方上傳組件, COS 組件來自於 oreilly cos 是一個高性能的文件上傳組件, cos 上傳組件是基於 Servlet 實現的, 這裏將 cos 組件拿過來剔除了 Servlet 依賴, 且刪除了和文件上傳無關的類. cos 組件可在源碼中 enoa-repeater-cos 獲取.

各服務器對 cos 上傳組件的支持

SERVER SUPPORT
FastCGI YES
Jetty YES
NanoHttpd YES
Netty YES
tio-httpserver NO
Tomcat YES

  • NanoHttpd

    cos 上傳也是 Repeater 中 NanoHttpd 唯一支持上傳的方式

  • tio-httpserver

    tio 因其特殊性, 無法支持 cos, cos 上傳通過 InputStream 進行寫入, tio 若要兼容就需要將 byte 數組轉化爲 InputStream, 這樣完全是浪費.

對於本身已支持文件上傳的提供者, 默認走的就是提供者自己的上傳方式, 若要切換爲 cos 上傳, 在 Repeater 啓動時提供配置告知即可, 下列表格是相應的服務器告知鍵.

SERVER KEY VALUE DESCRIPTION
FastCGI 無需提供, FastCGI 文件上傳必須使用 cos
Jetty provider.jetty.upload.vendor cos -
NanoHttpd 無需提供, NanoHttpd 文件上傳必須使用 cos
Netty provider.netty.upload.vendor cos -
tio-httpserver 不支持
Tomcat provider.tomcat.upload.vendor cos -

如何使用見 案例

Servlet

Repeater 對 Servlet 容器進行了部分改造, 使之與 Repeater 的 Request 機制相同.

  • 在 Servlet 中, Request Body 只能夠讀取一次, Repeater 中沒有此限制.
  • 文件上傳表單請求, 必須要先獲取文件, 才可獲取其他參數, Repeater 中沒有此限制.

這些改造是目前已支持的 Servlet 容器, 若有其他的 Servlet 容器接入, 也建議同樣對這部分進行改造.

FastCGI

FastCGI 並不是單獨啓動後直接訪問的, 而是藉助於支持 FastCGI 的 Web 服務器. 例如 Nginx.

假設 Repeater 使用 FastCGI 提供者啓動, 端口是 9100. 那麼可以在 Nginx 增加如下配置:

server {
  listen       80;
  server_name  localhost;

  location / {
    root           html;
    fastcgi_pass   localhost:9100;
    include        fastcgi_params;
  }
}

接下來請求 Nginx 的綁定的端口即可, 例如這個配置, 可以直接請求 localhost

案例

Repeater 的使用非常簡單, 下面是個最簡單的啓動 Repeater 代碼.

Repeater.createServer(new NanoHTTPDProvider())
      .listen(9100);

這樣就啓動了一個基於 NanoHttpd 的 Web 服務器.

當然, 這樣是沒有意義的. 接下來看看高級用法.

EoxConfig config = new EoxConfig.Builder()
  .debug(true)
  .info(true)
  .infoUseLog(false)
  .context("/example")
  .tmp(String.format("%s%stmp", PathKit.getRootPath(), File.separator))
  .other(Kv.by("provider.jetty.upload.vendor", "cos"))
  .build();

RepeaterFactory factory = new RepeaterFactory() {
  @Override
  public EoxAccessor accessor() {
    return new ExampleRepeaterAccessorImpl();
  }

  @Override
  public EoLogFactory log() {
    return new Slf4JLogProvider();
  }
};

Repeater.createServer(new JettyProvider())
  .ssl(false)
  .factory(factory)
  .config(config)
  .listen(9100);

這樣就啓動了一個 Jetty 的 Web 服務器.

這裏面涉及了兩個重要的方法: factory, config

  • factory()

    Repeater 啓動組件依賴

  • config()

    Repeater 啓動配置

factory 要求 RepeaterFactory, 這個之前已經介紹過了. 這裏來說說 config.

config 要求提供 EoxConfig, EoxConfig 中有如下配置

  • debug

    是否 debug 模式

  • context

    web 請求 context

  • charset

    編碼

  • info

    是否打印 Repeater 信息

  • infoUseLog

    Repeater 相關信息默認使用 System.out.print 打印, 如果這裏設置 true, 則會使用 Log 組件打印.

  • soTimeout

    請求超時時間

  • tmp

    文件上傳臨時存儲路徑

  • maxUploadSize

    文件上傳最大尺寸, 單位 MB

  • ssl

    如果設置爲 ssl 啓動, 此處是 ssl 的相關配置

  • other

    其他資源配置, 例如 cos 文件上傳告知鍵就放置與此

Repeater 的使用就這樣結束了, 更多豐富的功能直接實現 EoxAccessor 接口, 在 factory 中提供即可, 你完全可以基於 Repeater 創建一個自己的 Web 框架, Repeater 已經將不同的 Web 服務器統一輸出, 剩下要做的就只是實現一個訪問器.

更多案例, 可在源碼中 /enoa-example/example-repeater 獲取, 並且在 /enoa-test/repeater 中有測試腳本, 測試 Repeater 功能.

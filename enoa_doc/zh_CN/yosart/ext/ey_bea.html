拦截器

拦截器扩展

场景

控制器前置扩展是实现自 YmBeforeActionExt 扩展, 在默认路由中, 会在找到请求资源后 首先调用 BeaExt 接下来在进入到相应的资源.

BeaExt 的核心接口是 io.enoa.ext.bea.beaction.Interceptor` 该接是前置的运行接口, 所有的前置类均来自于此接口.

Interceptor

一个 Interceptor 使用案例

public class GoPartyInterceptor implements Interceptor {
  @Override
  public void intercept(Invocation inv) {
    YoRequest req = inv.req();
    Resp resp = inv.resp();
    resp.renderHtml("Interceptor test");
    inv.next();
  }
}

Interceptor 需要注意的一点是, 当实现了 Interceptor 接口, 在 intecept 方法中, 需要调用 inv.next() 才可进入到后续的 Interceptor, 如果不调用, 则会在此直接返回.

灵活运用 Interceptor 可以做很多事情, 例如可以使用 Interceptor 进行权限认证, 如果权限认证没过, 不继续向后走, 直接返回消息.

Handler

Handler 接口集成自 Interceptor, 与 Interceptor 的区别是, Handler 不需要调用 next 方法, 该方法被 Handler 进行了封装, 实现 handle 方法即可.

public class PartyGoHandler implements Handler {
  @Override
  public void handle(YoRequest req, Resp resp) {
    resp.header(new Header("key", "value"));
  }
}

Handler 的应用也很广泛, 例如完全可以利用 Handler 设计一个 session 管理器.

Validator

Validator 实现自 Interceptor, 功能是对请求参数进行验证, 如果未通过验证则直接返回消息, 此功能是基于 Interceptor 的封装, 用 Interceptor 接口也同样可实现此功能.

public class PartyGoValid extends Validator {

  @Override
  protected boolean shortCircuit() {
    return true;
  }

  @Override
  protected void valid(YoRequest req) {
    String valid = req.para("valid", "pass");
    if (!"pass".equals(valid))
      super.error("msg.valid.pass", "Can not pass.");
  }

  @Override
  protected void handle(Kv attr, Resp resp) {
    resp.renderHtml(JsonKit.toJson(attr));
  }

}

Validator 主要有三个方法valid , error 以及handle, valid 方法用于进行参数验证, 如果验证失败可以调用error 方法, 传递错误消息, 当鉴定有错误后会进入到handle方法, 进行信息渲染.

shortCircuit 短路配置, 默认为 true, 在 valid 方法验证时, 是否遇到第一个 error 调用就跳出.

Before

Before 注解是拦截器使用的载体, 在方法的签名上使用 Before 注解即可使用拦截器.

Control

  @Before({GoPartyInterceptor.class, PartyGoValid.class, PartyGoHandler.class})
  public void go() {
    super.render("go");
  }

Action

new YoAction() {
  @Before({ActionAnnoValid.class, PartyGoHandler.class})
  @Override
  public Response handle(YoRequest request) {
    return Resp.with(request).render("action anno").end();
  }
}

Clear

Clear 注解, 用于清除拦截器, 最常见的场景是, 账户登录设置如果使用拦截器实现, 在登录/注册界面是不需要进行拦截的, 但是其他页面都需要, 这时可将拦截器设置为全局拦截器, 而登录注册则使用Clear 清除该拦截器.

@Clear(LoginInterceptor.class)
public void login() {
  super.render("login");
}

使用

Global

全局拦击器, 如果一个拦截器被设置为全局为全局拦截器, 那么每个请求都会进入到该拦截器, 全局拦截器的注册稍有不同, 并非通过注解使用, 而是在创建BeaExt时指定的.

new BeaExt().global(new GlobalHandler())

Control

Control 类型拦截器, 需要放置与 Control 方法签名上.

Action

Action 同样支持该拦截器, 需要放置于 handle 方法签名上.

细腻化控制

相对而言, 只提供Global, Control, Action 三种拦截方式并不全面, 而且Before 以及Clear 注解限制只能在方法签名, 不可在类的签名上. 因此无法做到细腻化的配置, 例如只给账户模块增加不同的拦截器, 等等.

后续将会支持该功能.

POM

<dependency>
  <groupId>io.enoa</groupId>
  <artifactId>yosart-ext-bea-beaction</artifactId>
</dependency>

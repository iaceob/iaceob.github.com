其他信息

JFinal

在这里特别说以下与 JFinal 的关系, 包括从源码中以及使用方式上, 有很多都和 JFinal 相似, 特别是在 toolkit, json 这些基础模块中, 很多都是从 JFinal 中搬运过来的, 实际上 JFinal 也是一个很简单的框架, 不过随着发展变得东西越来越多, 但是工程中这些是用不到的, 在我个人的想像中这些应该都是可选项, 应该在我想要的时候出现, 而不是打包一个全家桶.

但是, 要特别声明的是, 虽然很多任务具类都是从 JFinal 中搬运过来, 甚至 Yosart 的默认路由实现都和 JFinal 极其相似, 实际上内部实现原理已经和 JFinal 有很大差别, 就只是使用方式大体相同而已, 之所以 Yosart 默认路由器会这么做, 最主要的考虑是 JFinal 的一些接口使用方式都相对而言非常简单.

只要你想, Yosart 可以长成任何模样, 甚至是 Spring 的方式, 因为所有都是可配置实现的.

OkHttp

Enoa 的源码中有很多数据类, 像是模板的配置类, DB 配置类 等等, 创建方式均借鉴自 OkHttp 的类创建方式, 使用 Builder 创建, 无 setter 方法. 避免了数据遭到篡改问题.

且 Repeater 中, 部分代码来自于 OkHttp, 例如 Cookie

命名

在 Enoa 的整个源码中充斥着大量的以 _ 开头命名的类, 以及无修饰符类, 特别是在 Yosart 源码中, 很多以 Yo Ya Ye Ym Oy Ay 开头的类.

这只是一个编码洁癖而已, 或者说 Enoa 的所有 接口, 代码 都是被精心设计的. 主要原因有以下几点:

首先以 _ 开头的类, 先列出以下几个

  • _FreemarkerEngine
  • _FreemarkerTemplate
  • _RepeaterPlusRequest
  • _RepeaterPlusRequestContext
  • _Json

_FreemarkerEngine, _FreemarkerTemplate 这两个类都是第三方的具体实现类, 别且无修饰符

_RepeaterPlusRequest, _RepeaterPlusRequestContext 这两个接口是 Repeater 接入第三方 Web 服务器时, 对第三方 Request 的增加接口, 修饰符为 public

_Json EnoaJson 的父类, 无修饰符

意图很明显

第一个, 因为是第三方的实现类, 与 enoa 无太大关系, 用以区别, 同类型的很多, 例如 _NanoApp _Jetty 等等.

第二个, 虽然是也是提供给第三方实现用, 不过是 enoa 体系中的接口, 而且修饰符是 public, 而在正常开发中, 这些接口是不可能被用到的, 只在进行第三方接入的时候才会用到, 不想在应用开发中, IDE 打出 Repeater 时会将这些接口也列出来, 造成选择障碍.

第三个, enoa 体系中的接口, 之所以提出来, 因为 EnoaJson 是一个抽象类, 有自己的代码要实现, 如果接口放在抽象类中会很乱, 因此就提出来.

Yosart 中的命名也是相同, 一句话总结就是: Yo 开头的类全部都是应用开发时会用到的类, Ya 开头的都是父类, Oy Ay 这些开头都是开发中几乎没有机会用到的类.

此外 Yosart 也没有占用很常用的名称, 包括 Control Action, 这些交给开发者自己去使用.

风格

链式调用

Yosart 的代码风格偏向与链式调用, 从 Repeater 以及 Yosart 中就可以看出, 直接一个链式全部完成.

getter&setter

无 getter setter 方法, enoa 几乎所有的代码都没有 getter setter 方法, 而是直接使用与变量名同名的方法.

String val = request.attr("key"); // 获取 attribute

request.attr("key0", "val").attr("key1", "val"); // 设置 attribute

但是这样也带来了一定的风险, 太多任务具, 特别是 json 工具类都是通过 getter setter 方法获取或者写入的, 在 Request 中就遇到了这个问题, 因此 Yosart 中特别增加了一个 data 方法来弥补.

无配置文档

无配置文档, enoa 的所有工程都不需要配置文档, 一切都用代码去完成, 忘掉配置文档, Java 的笨重很大程度上是早期的一些框架带的风气, Spring Struts Hibernate 之流, xml 配置写到怀疑人生, 还很难以理解, 文档出的一套一套.

少注解

目前, 整个 enoa 体系中, 受到直接支持的注解仅有 Yosart 中的 Action 注解, 用于路由解析. 其他的都交给扩展去做. 之所以这么做, 是因为不希望看到 Yosart 会出现向下面类似的代码:

@ApiOperation(value="Get user list", notes="Get user list", response = Message.class)
@ApiImplicitParams({
  @ApiImplicitParam(name = "page", value = "Page No.", required = false, dataType = "int", paramType = "query", defaultValue = "0"),
  @ApiImplicitParam(name = "size", value = "Page size", required = false, dataType = "int", paramType = "query", defaultValue = "20"),
  @ApiImplicitParam(name = "sort", value = "Sort rule", required = false, dataType = "String", paramType = "query", defaultValue = "id,desc")
})
@ApiResponses(value = {
  @ApiResponse(code = 0, message = "Success！"),
  @ApiResponse(code = 1, message = "Fail！")
})
@RequestMapping(value="/list", method = RequestMethod.GET)
public ResponseEntity<?> list(@RequestBody UserSearch userSearch, 
                              PagedResourcesAssembler<User> assember, Pageable pageable){
  logger.info("API start > user list. paras：{}", memberSearch);
  Page<User> page = userService.list(userSearch, pageable);
  logger.info("API end > user list. returned：{}", page);
  return new ResponseEntity<>(new Message(Result.SUCCESS, assember.toResource(page)), HttpStatus.OK);
}

要多恶心就有多恶心的代码, 整个这么长的代码有用的就一行, 注解比正常代码还多, 尽管是因为有 swigger 的原因 (enoa 的文档生成以及继承测试后续也会添加).

来看看实际有用的代码

@RequestMapping(value="/list", method = RequestMethod.GET)
public ResponseEntity<?> list(@RequestBody UserSearch userSearch, 
                              PagedResourcesAssembler<User> assember, Pageable pageable){
  Page<User> page = userService.list(userSearch, pageable);
  return new ResponseEntity<>(new Message(Result.SUCCESS, assember.toResource(page)), HttpStatus.OK);
}

这才是 enoa 想到达到的效果, 一大堆注解不是 enoa 想要的.

注解也是那些从前一大堆 xml 的框架觉得确实有点麻烦而找到的替代的方案, 然而无论怎么改都逃脱不了难看的宿命, 他们弄的太复杂了, 很简单的事情.

最后

如果你觉得简单的 enoa 并不好, 没有关系, 在 Yosart 方面, 完全可以根据路由扩展实现你想要的, 甚至直接基于 Repeater 来实现一个. 当然 也是可以继续去用你习惯用的.

enoa 会继续简单下去, 并且不断的完善.

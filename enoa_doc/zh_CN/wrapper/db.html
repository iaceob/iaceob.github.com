数据库

数据库

数据库组件主要分为两部分

  • 连接池
  • 数据库操作

这两者都有不同的实现, 可以进行自由选择搭配.

连接池

目前支持的连接池有

  • c3p0
  • druid
  • hikaricp

连接池在数据库体系中没有被设计成单独启动, 那是没有意义的, 连接池的最终目的是获取 DataSource, 连接池的启动在数据库的启动时会自动启动, 只要提供相应的配置即可. 但是, 这并不是绝对的, 在 enoa 中所有组件的自由度都很高, 没有强制的关系以来, 如果想要单独启动连接池接入到其他体系中也很简单.

例如这里单独启动一个 c3p0 数据库连接池

EnoaDs ds = new C3p0Provider().dataSource(
  new C3p0Config.Builder()
    .driver("com.mysql.jdbc.Driver")
    .url("jdbc://xxx")
    .user("root")
    .passwd("pwd")
    .build()
  );
DataSource dataSource = ds.open();

但是在 enoa 中, 无需启动, 具体使用见后文.

连接池中还有一个需要注意的点是连接池的配置信息. 目前提供的连接池组件都有单独的配置类.

POOL CONFIG CLASS
c3p0 io.enoa.db.provider.ds.c3p0.C3p0Config
druid io.enoa.db.provider.ds.druid.DruidConfig
hikaricp io.enoa.db.provider.ds.hikaricp.HikariCpConfig

这些配置类都是统一实现 io.enoa.db.EoDsConfig 接口的, 因此连接池的启动可以不一定比需要相应的 config 只要是实现 EoDsConfig 接口类即可. 例如上方的 c3p0 连接池启动, 可以使用下面的方式.

EnoaDs ds = new C3p0Provider().dataSource(new EoDsConfig() {
  @Override
  public String driver() {
    return "com.mysql.jdbc.Driver";
  }

  @Override
  public String name() {
    return "root";
  }

  @Override
  public String url() {
    return "jdbc://xxx";
  }

  @Override
  public String user() {
    return "root";
  }

  @Override
  public String passwd() {
    return "pwd";
  }
});
DataSource dataSource = ds.open();

这两者的效果是相同的.

EoDsConfig 只是将连接池共用的参数提取出来, 在使用的时候转换成自己能识别的配置, 如果不是能识别的进使用 EoDsConfig 共用接口, 其他使用默认值. 之所以针对不同连接池提出不同的配置文档, 是不同的连接池有不同的配置, 如果需要则使用.

数据库操作

数据库操作, 目前支持

  • ActiveRecord
  • BeetlSQL
  • Mybatis

注意: 这里不打算详细介绍不同的资料库操作框架如何使用, 如需要请到相应的框架文档去了解.

下方资料库的所有代码仅是示例, 可从 enoa 源码中 enoa-example/example-db/src/test/java/io/enoa/db/EnoaDbMgrTest 中获取.

ActiveRecord

ActiveRecord 是由 JFinal 提供的数据库操作方案, 使用方式:

// main ActiveRecord
EoDbConfig dbc1 = new ActiveRecordConfig.Builder()
  .baseSqlTemplatePath(PathKit.getRootPath().concat("/src/test/sqls/activerecord"))
  .sqlTemplate("template.sql")
  .dialect(new MysqlDialect())
  .ds(DS.C3P0_MYSQL.ds, DS.C3P0_MYSQL.config)
  .build();
EnoaDbMgr.me().start(new ActiveRecordProvider(), dbc1);
List<Record> rets1 = Db.find(Db.getSql("User.list"));

// pgsql ActiveRecord
EoDbConfig dbc2 = new ActiveRecordConfig.Builder()
  .name("pgsql")
  .baseSqlTemplatePath(PathKit.getRootPath().concat("/src/test/sqls/activerecord"))
  .sqlTemplate("template.sql")
  .dialect(new MysqlDialect())
  .ds(DS.HIKARICP_PGSQL.ds, DS.HIKARICP_PGSQL.config)
  .build();
EnoaDbMgr.me().start(new ActiveRecordProvider(), dbc2);
List<Record> rets2 = Db.use("pgsql").find(Db.getSql("User.list"));

上方的案例提供了连接两个数据库的案例, 只需这样即可启动 ActiveRecord, 使用方式遵循 ActiveRecord 提供的 API 即可.

ActiveRecord 在 JFinal 中并非是单独的存在, 而是集成在 JFinal 整个体系中, 但是相对来说这不符合 enoa 的初衷, enoa 的目的就是将这些剥离出来不做强依赖, 一切都是组件插入进去.

因此, 在 ActiveRecord 这方面, 有从 JFinal 中提取一个独立的 ActiveRecord.

https://github.com/iaceob/activerecord

提取版本从 JFinal 3.3 中, 后续更新不一定能和官方保持同步, 实际使用建议依赖 JFinal 保证使用的是最新且安全的.

BeetlSQL

使用案例:

// mysql test
EoDbConfig dbc1 = new BeetlSQLConfig.Builder()
  .load("/beetl")
  .ds(DS.DRUID_MYSQL.ds, DS.DRUID_MYSQL.config)
  .style(new MySqlStyle())
  .build();
EnoaDbMgr.me().start(new BeetlSQLProvider(), dbc1);
List<Map> rets1 = BeetlSQLKit.select("User.list", Map.class);


// postgresql test
EoDbConfig dbc2 = new BeetlSQLConfig.Builder()
  .name("pgsql")
  .load("/beetl")
  .ds(DS.C3P0_PGSQL.ds, DS.C3P0_PGSQL.config)
  .style(new PostgresStyle())
  .build();
EnoaDbMgr.me().start(new BeetlSQLProvider(), dbc2);
List<Map> rets2 = BeetlSQLKit.use("pgsql").select("user.list", Map.class);

BeetlSQL 在 enoa 中, 添加的多数据库源的支持, 因此如果想要获取 SQLManager 通过 BeetlSQLKit.use 进行获取.

Mybatis

使用案例:

EoDbConfig dbc1 = new MybatisConfig.Builder()
  .scan("io.enoa.db.mapper", BaseMapper.class)
  .mapper(PathKit.getRootPath().concat("/src/test/sqls/mybatis"))
  .suffix("xml")
  .ds(DS.HIKARICP_MYSQL.ds, DS.HIKARICP_MYSQL.config)
  .build();
EnoaDbMgr.me().start(new MybatisProvider(), dbc1);
List<Map> rets1 = MybatisKit.with(UserMapper.class).list();

EoDbConfig dbc2 = new MybatisConfig.Builder()
  .scan("io.enoa.db.mapper", BaseMapper.class)
  .mapper(PathKit.getRootPath().concat("/src/test/sqls/mybatis"))
  .suffix("xml")
  .name("pgsql")
  .ds(DS.DRUID_PGSQL.ds, DS.DRUID_PGSQL.config)
  .build();
EnoaDbMgr.me().start(new MybatisProvider(), dbc2);
List<Map> rets2 = MybatisKit.use("pgsql").with(UserMapper.class).list();

Mybatis 在 enoa 中, 抛弃了 xml 方式配置加载, 配置文档加载不是 enoa 理想的加载方式, 而且, 见过 enoa 的启动方式后, 相信你也会觉得 xml 的配置是有多么的繁琐, 原在 xml 中的所有配置均可在 MyatisConfig 中实现. 并且添加了多数据源的支持, 获取 SqlSession 通过 MybatisKit.use 方法, 并且对 SqlSession 的使用进行了一次封装, 直接调用 with 即可获取到 mapper.

语法分析

从上方给出的案例代码能看出, 启动一个数据库操作框架, 直接调用 EnoaDbMgr.me().start 方法即可, 传入想要的提供者, 并且提供该提供者需求的配置信息.

关于配置方面, 需要注意的是, ds 配置是必须要有的, 该配置决定了要启动的数据库操作框架使用哪个连接池. 连接池完全可以自由搭配, 看实际情况选用. 连接池如何使用见本章第一部分 连接池

EnoaDbMgr 也并不是必须的, 如果不想使用 EnoaDbMgr 启动, 也完全可以直接使用提供者自身启动, 例如 Mybatis 可以通过下面的方式: 

EoDbFactory<MybatisConfig> factory = new MybatisProvider();
EnoaDb db = factory.db(new MybatisConfig.Builder()
  .name("main")
  .ds(new C3p0Provider(), new C3p0Config.Builder()
    .driver("com.mysql.jdbc.Driver")
    .url("jdbc://xxx")
    .user("root")
    .passwd("pwd").build())
  .scan("io.enoa.db.mapper", BaseMapper.class)
  .mapper(PathKit.getRootPath().concat("/src/test/sqls/mybatis"))
  .suffix("xml")
  .build()
);
db.start();

数据库启动配置数据几乎无法做到通用, 相同的只是都需要连接池, 因此不同的提供者提供了不同的配置类, 现有的提供者配置类如下

DB CONFIG CLASS
ActiveRecord io.enoa.db.provider.db.activerecord.ActiveRecordConfig
BeetlSQL io.enoa.db.provider.db.beetlsql.BeetlSQLConfig
Mybatis io.enoa.db.provider.db.mybatis.MybatisConfig

最后, 请记住, enoa 对数据库的封装只是简单的将不同框架的实现统一方式处理而已, 原框架怎么用 enoa 接入后也是一样的用法, 包括启动配置等等.

开发

如果现有的连接池或者数据库操作组件并不是你想要的, 想要接入其他的实现也很简单.

连接池

与连接池相关的接口:

  • EoDsFactory
  • EoDsConfig
  • EnoaDs

EoDsFactory 要添加的提供者接口, 该接口要求返回 EnoaDs 也就是该提供者的具体实现, EoDsConfig 是该提供者要求的配置数据.

调用关系

EnoaDs = EoDsFactory.dataSource(EoDsConfig)

数据库操作

与数据库操作相关的接口:

  • EoDbFactory
  • EoDbConfig
  • EnoaDb

EoDbFactory 要添加的提供者接口, 该接口要求返回 EnoaDb 也就是该提供者的具体实现, EoDbConfig 该提供者要求的配置数据.

调用关系

EnoaDb = EoDbFactory.start(EoDbConfig)

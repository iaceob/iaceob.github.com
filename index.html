Request InputStream 首先獲取

事情是這樣, 因為需要文件上傳功能, 就按照以前使用過的方式在控制器中使用 getFile 獲取上傳的文件, 但是這次報了如下錯誤

從報錯信息中沒有看出是什麼地方的錯誤, 起初懷疑是在文件上傳表單中未設置 multipart/form-data; 而導致無法獲取文件, 遂排查, 但發現明明是有設置, 並且在 Request Header 中確實是 multipart/form-data; , 工程中關於文件上傳的依賴包也已經引入.
那問題就不是發生在這一步, 後來在控制器中輸出 request.getContentLength 發現是有長度, 那麼也確實文件是有被上傳的.

那麼問題就是發生在 getFile 方法中, 定位到錯誤地方, 進入調試最終定位的錯誤的代碼是 org.eclipse.jetty.server.Request

在使用 getFile 獲取文件時, 發現 this._inputState 的值是 2, 所以就不滿足這個 if 判斷, 而直接拋出 IllegalStateException 異常.

所以就得找 this._inputState 是在何處被賦值為 2 .

後在 Request 類中的 getReader 方法中發現

原來在此處! 問題就是發生在這裡, 因為這次項目中提供的接口中參數傳遞都是使用 JSON 方式, 為了便於在控制器中解析數據, 遂加了一個參數解析的過濾器, 將獲取到的 JSON 字符串轉換為對象以便於在控制器中取值, 獲取 JSON 時就調用了 getReader 方法獲取流中的 JSON 數據, 這是執行在全局的過濾器, 所以在後續的控制器中通過 getFile 也就是 getInputStream 中獲取上傳的文件時就會拋出前面提到的錯誤.

so, 文件上傳的控制器中取消參數解析的過濾器即可, 獲取 getFile 必須是在獲取參數之前的, 否則將無法獲取.

起點

翻翻日曆(應景), 臥槽都特麼四月了 !!!!
這一年都特麼到了三分之一了 !!!!
從來沒感覺過能有這麼快的啊 !!!!

日常還是日常, 似乎仍然沒有收拾好心情來做事.

這期間值得一提的事情是, 第一本主動閱讀的書讀完了 《消失的江城》 從去年 (2015) 直到前幾天清明節假期回家路上讀完, 說來也巧, 書中有一章節是 春節 , 而剛好是我在春節回家途中開始閱讀的, 又是春天 又剛好是在這剛步入到春季時閱讀, 剛好恰到好處的感受了一番, 確實是一本有意思的書籍.

so, 下一個書單起點開始了 《甲骨文》

然後,

開始準備向後面開始準備吧, 可能不久就會離開吧 ╮(╯_╰)╭ 仍未發現哪裡是可以停留的地方, 流浪吧.

這幾次回家都和爺爺深談了好久, 談以前的家鄉, 談以前爺爺小時候, 用滿滿的好奇去問爺爺很多問題,

比如以前的黃尾鎮, 容我畫張圖表達試一試

原諒我放蕩不羈的畫風

故事是這樣的, 爺爺是在 1950 年出生的共和國第一代國名, 那個時代的黃尾大概就像這樣 (雖然似乎直到現在也基本還是這樣), 在河的這面說是城牆 (聽的我一陣驚訝, 我說怎麼現在城牆沒了了!!), 爺爺那會還和小夥伴過城門到河裡去游泳.
在 60 年還是哪一年來著, 發大水把整個城牆給沖走了, 整個黃尾鎮的房子都被沖走了, 我是各種不相信啊, 竟然這麼誇張, 而且城牆還給沖走了, 那可是城牆欸.
再細一說, 原來所謂的城牆是攔河壩, 比地稍微高一些, 對嗎 這樣才合乎情理啊, 本來家鄉這都可以算得上是荒無人煙的地方了, 怎麼會又城牆這麼高大上的東西, 在那沖毀了之後, 這個攔河壩又重修了, 一直到現在, 現在這個壩比以前還要長

然後是這座橋, 容我在畫一張圖

當然這麼大的水肯定也是把整座橋給沖毀了的 (未知是否那次沖毀後重建的新橋), 以前的橋只有一人能走的寬度, 十三節, 每節多長來著 (說著我給忘了, 注意! 圖上的也是十三節), 那時候還有乘船過河的, 河道很寬, 還說這座橋有過人走時不慎摔到河裡死人的.
之後就是這座新建的橋了, 橋很寬, 得有 7 米吧應該 (應該差不多, 可能不止), 石拱橋, 三拱, 但拱得又超過 20 米 (感覺而已), 佩服那時勤勞的人, 修了這樣一座橋, 真的, 這座橋很漂亮, 全部都是石結構, 有了這座橋之後, 沒有了划船的人, 也見不到船了, 當然就算有船也已經划不過去了, 並不是沒人會划船, 而是共和國決定在河道的上方修建了一個水電站, 以及不知道什麼原因河道的水也是慢慢的變少了, 自此之後, 正常情況下, 這條河的河水的水位就下降了, 就連河中間的河床都基本浮現了出來, 只有很小的一部分流淌這湍急的河水. 再之後, 這幾年在河道上游, 水庫下游又給修建了一個旅遊區, 又給加了一道水壩, 水量又一次減少, 我覺得這似乎並不是一件好事, 大好環境就被如此人為的給破壞了, 已經回不到最初的河流了.

被破壞的又何止只是這條河流了.

原本堅挺有力的山峰, 在山頂又被某個開發商在那採掘石頭, 採掘後的石頭又直接在河邊二次加工, 加工後的乳白色廢水排放在河里, 在往下走一個支流上方是一個鐵礦加工區, 排出黃色廢水, 結合後黃白相間向下游流去, 甚至那鐵礦加工區還直接改變原有彎曲河道從一個伸出的半島直接開口, 而那彎曲的部分放置礦渣. 現在鐵礦開採也沒有在繼續了, 這個鐵礦留給家鄉的就三點:

  • 一夜暴富的移民戶
  • 未被善後的採礦坑
  • 被礦渣填平的河道

各種被破壞的現狀.

說遠了

然後就和爺爺說了句, 以前真好, 回應我的是一句:

「好個屁, 以前都吃不飽, 那個時候很苦.」

當然, 我已經很難體會那時候了, 只能說略有點感受吧, 至少在我的記憶中, 以及我所經歷的見聞, 以前確實是很清貧. 但是以這樣的環境犧牲為代價是可取的?

這次下來還聽到爸爸這麼說了:

「以前看山上到處都是花, 能問到各種花香, 而現在山上的花都少見了, 映山紅 蘭草 都沒了」

在這次之前的一次談話哩, 說了一說以前爺爺小時候發生的事情.

因為爺爺是 1950 年出生的, 所以了, 共和國成立之初開始摸石頭過河做過的路爺爺可以說是個盡職的摸石頭的人, 練過鋼 當過紅衛兵 吃食堂 各種計劃經濟票據, 談到批鬥一事, 說是那個時候整天不做事情, 就是跟著隊伍從這跑到那響應號召. 而且有真正見過人被批鬥, 說是很殘忍以至於我想繼續問下去但是卻想了好久才慢慢道出來, 這也許是最難以回首的經歷吧, 談過之後才瞭解到原來處在身邊的人當時都有這樣的經歷甚至是被批鬥的一方.
政治甚至能影響到這樣的偏僻小山村, 在那個道路不通, 消息敝塞的時代, 真真切切的印象了那一代人.

好吧, 就說這些了, 這也不適談的太深. ╮(╯_╰)╭

基本上了, 每次都是我先終止了話題, 沒有在繼續談下去, 也不知是在某個時刻產生的想法, 很想已書籍的形式來記錄這些, 但是又無從下手, 關於那個年代的故事我們現在的人已經知之甚少了, 或者說能知道的渠道基本都不存在, 而像爺爺奶奶這樣年齡的人又會慢慢的少去, 記憶改如何留存下去了 ? 可惜我並不很擅長寫文章.

好啦, 四月開始了, 新的起點又來了.

寶寶的生日也快到了.

生活總是要面對的, 最後說說最近學的新知識吧

沒錯 還是寶寶教的 XD

然後

勉強能認識這些 _(:3 」∠ )_

在 Qt 中一個重要的概念就是信號與槽機制, 就是這個機制來實現用戶界面與後臺邏輯對接, 一個簡單的實例如下:

信號與槽的連接通過 QObject::connect 來連接, 有四個參數

實現的方式就是某個發射源發射一個信號交給一個接收信號體去接收信號, 在信號實現中做具體實現.
例如上方, 由 ui 中的登入按鈕 (this->ui->btnLogin) 發射點擊信號(事件) (SIGNAL(clicked(bool))) 由當前類接收 (this) 使用 loginSlot 槽做邏輯實現 (SLOT(loginSlot())).

數據庫設計中層級關係數據設計

常規型

層級關係會出現在絕大部分系統中, 通常的做法 (也可以說是標準做法) 如下:

id pid name
1 0 home
2 1 linux
3 2 ubuntu
4 1 develop

上述表中所要描述的是如下關係

使用 pid 字段來達到描述層級關係的目的, 但是這種設計方案所帶來的問題:

優點

  • 層級關係維護簡單
  • 擴展性強

缺點

  • 數據不直觀

    如果出現相同名稱 (在設計系統節點時, 不同控制器下可能會出現相同的路由名, 例如: save delete 等), 不容易判斷出當前記錄輸入某個父模塊; 另就算不重名, 也很難根據某一條數據顯示出此條記錄的層級信息.

  • 數據結構構造複雜

    通常需要使用遞歸來實現數據結構分析 (代碼中或者數據庫遞歸).

不要命型

就算死也要死的華麗.
這是一種簡單暴力的方式實現層級關係

一個簡單的數據結構如下, 數據可以美麗成這樣:

id path name
1 0 總部
2 0-1 天朝分部
3 0-1 美利堅分部
4 0-1-2 河北
5 0-1-2 河南
6 0-1-3 華盛頓

美麗到一眼就能看出來想要表達什麼意思.

總部的自標為 0
然後下一級, 天朝分部和美利堅分部, 則繼承總部的 0 為表忠誠, 把上級大旗拿來標榜, 遂變成 0-1
接下來到在下面, 河北 河南 華盛頓 把上級的稱謂拿出來, 為表忠誠, 把上級大旗拿來標榜, 遂變成 0-1-2 與 0-1-3

呵!呵!呵!

好, 現在總部想看看到底有多少手下, 哪些手下.

那麼 就可以這樣
例如在 SqlServer 中就可以使用此種方式進行查詢, 並比較結果

id relpath name
1 0-1 總部
2 0-1-2 天朝分部
3 0-1-2-3 河北
4 0-1-2-4 河南
5 0-1-5 美利堅分部
6 0-1-5-6 華盛頓

也就是把每個人自己的旗幟放到自己的等級上, 然後就能按照這個等級來排個序.

perfect!

好 嚴肅

雖然簡簡單單的就實現了層級關係, 但是這有個非常嚴重的問題!
公司發展的越來越好, 總部一直在想著要換一個更顯地位的地方, 但是原來的地還留著, 處理一些閒雜事, 那這個時候總部搬遷, 新增了個標識是 93204 的總部, 等級仍然是最高 (0) , 這就面臨著所有分部都得將總部的標識給改掉, 然而, 這並不好改.

問:
如何將
0-1 0-1-2 0-1-2 0-1-3
改為
0-93204 0-93204-2 0-93204-3

所以, 這並不是一個很好的方案.

優點

  • 數據直觀
  • 數據結構構造簡單

缺點

  • 不易於調整
  • 擴展性差

中和型

其實就是 不要命型 的翻版, 換湯不換藥

在說這之前, 先看一段數據

這看起來是不是就很像了, 此處的一串數字是地區以及下轄地區的獨立編碼
下轄的編碼從上級的編碼從末尾開始按順序排, 這樣就實現了多層級關係.

然而, 一個大的問題是, 例如北京市轄區 110100 從現有的數據看, 市轄區最做只能容納 100 個區域, 多了就沒法放 (前提是真有 100+ 個區)

在特殊場景的應用效果很好 (例如地區編碼)

優點

  • 數據直觀, 簡單明瞭, 可讀性高
  • 數據結構構造簡單
  • 便於維護， 方便查找以及記憶

缺點

  • 擴展性差
  • 不易於調整

總結

常規型 不要命型 中和型
可擴展 Y N N
易構造 N Y Y
易讀 N Y Y
易改 Y N N
效率 (開發) ↓ - ↑
場景 所有場景 所有場景 (前提是不怕死) 特殊

吐槽

自說要寫個博客開始, 都已經有幾年時間了, 可是到現在仍然沒寫完 ( 實際中已經有個測試版已經上線: atlantismonk ), 其實哩整個功能也都已經完成了, 可最終被我給拋棄了 :( 就是覺得弄的不夠好, 可是又沒動力去改了 ╮(￣▽￣)╭ 老毛病嘛, 得改, 犯了不知多少次了每次也都說要改, 每次又都沒改 (┙>∧<)┙へ┻┻ 就像寫這篇文章, 從 2 月還沒結束的時候就想寫了, 那會想要選個好域名什麼的, 但是… 欸, 腦漿不夠多並不能想到什麼好域名, 最後無奈拿了這 (3u3) , ← 看這貨, 是不是挺有意思的 XD (強行括號) 在選這之前其實也試了好多個, 像 olo 0l0 oolloo o0o0o 可是特麼都被別人給搶走了 (這些人肯定和我想法一樣)

所以就直接切到 hexo 上來了, 說道這個 hexo, 是一個叫 @tommy351 的臺灣人寫的, 說到這想要說的是, 因爲我常用 [繁體字 | 正體字] 而被身邊人各種吐槽:

你是香港來的嗎, 用什麼繁體字
我讀書少, 不要打繁體字
…

= = 無力吐槽, 也不知爲什麼我怎麼就能夠看懂, 自帶翻譯器, 肯定是了, 雖然我不反對簡體字(假的), 不過我總感覺不能接受繁體字是一種忘本的表現, 你簡體再牛逼在好看不也特麼是從用了幾千年的繁體字演變過來的, 在我的接觸中除了大陸會明顯的表示對繁體的厭惡(大部分), 而像東南亞這些官方認簡體的人接觸後發現, 並不反對繁體, 甚至是主動繁體字, 當然啦也不能全盤否認咯, 是簡是繁需要用時間來驗證(前些天看到了一篇非常不錯的文章, 可惜當時沒保存, 待日後看到再補上).
so 因爲環境所限, 可能以後寫的文章有非常大的可能是出現簡體版, 又會有繁體版本, 甚至簡繁混搭 (儘量, 基本不會有).

最後吐槽下, 從今年工作以來一個多月了卻沒做什麼事情 (￣□￣) (我應該高興嘛), 博客已經放棄弄了, 那爬蟲也基本穩定了, 可是這沒有結束, 已經確定! 這的整個系統都得改, 全盤改! 我是見過局部改動, 最大不過改改數據庫幾個表什麼的, 這裏的改法也還真是奇耙 = =, 當然, 這也不是沒有原因吶, 數據庫設計算的上是軟件的靈魂設計, 骨架構建, 會出現這樣的原因當然問題就出在數據庫上 (公司 DBA 年前已離職, 自那時起多了個兼職 DBA 的角色), 面對前不知多少個拿到證的 DBA 設計的這個數據庫, 真的是叫一個大寫的 XX , 呵呵.
結果就是一改的話, 前面幾年所做的東西都得跟着一起改 ╮(￣▽￣)╭ 改吧 ~

學習

自這個年度開工以來, 由於各種閒, 開始折騰起 Forevermark 的那系統 (以前的爛系統, 各種變需求), 現在在去弄反而沒有了在職時的種種限制, 完全可以根據喜好來弄 (ง •̀_•́)ง 之前昂, 單獨做了一個服務, 定時執行, 因爲單就 WEB 的話, 因爲種種庫存操作比較慢, 導致數據提交 HTTP 請求超時, 而導致數據提交失敗, 而單獨服務雖然能夠解決這個問題, 但是隨之而來的又導致了另一個問題, 數據延時太長, 而導致系統的庫存不準確造成更大的問題 (雖然現在還沒發生, 發生就事大了)
so 決定使用客戶端來解決, 可是我又不想用 Java 來寫 (其實也不會), 在一個我用的是 Linux 系統, 而使用的人又都是 Windows 那就決定這個必須得跨平臺, 最後選擇了以前就用過 (想學) 的 Qt 語言 (第一次接觸是在 Ubuntu 那年準備出手機, 感覺挺有意思, 準備學習學習, 最終學習的解決也和 Canonical 想出的手機一樣, 失敗了, Canonical 給自己定了一個很高的目標 我也給自己定了一個很高的目標, 但是 Canonical 至今仍然在堅持, 而我早已放棄), 所以就這麼搞起來了.
話說啊! 現在已經能夠作出一些很簡單的案例了 XDD 大概能理解了信號和槽的機制 (和 JS 的事件綁定真的很相似).

另還是想在認真看看 NodeJS, 對於 NodeJS 或者說 JS, 還是比較喜歡的.

讀書

欸, 這個話題叻, 其實從扔書本之後就從沒認真看過書 (其實就算沒扔那會也壓根就沒怎麼看過), 可是不知是從什麼時候開始, 很有可能是見到了這個人說的 (我自己都想不起來是怎麼覺得想看看書的):

阅读一百本计算机著作吧，少年

很多人觉得自己技术进步很慢，学习效率低，我觉得一个重要原因是看的书少了。多少是多呢？起码得看3、4、5、6米吧。给个具体的数量，那就100本书吧。很多人知识结构不好而且不系统，因为在特定领域有一个足够量的知识量+足够良好的知识结构，系统化以后就足以应对大量未曾遇到过的问题。

奉劝自学者：构建特定领域的知识结构体系的路径中再也没有比学习该专业的专业课程更好的了。如果我的知识结构体系足以囊括面试官的大部分甚至吞并他的知识结构体系的话，读到他言语中的一个词我们就已经知道他要表达什么，我们可以让他坐“上位”毕竟他是面试官，但是在知识结构体系以及心理上我们就居高临下。

所以，阅读一百本计算机著作吧，少年！
